## 一、实习经历：Agent 交互式专利检索系统

### 1. 三级 Fallback 意图识别流水线

**Q1：为什么要设计三级，而不是直接用大模型分类？**
【意图】考察方案设计的权衡思维，是不是真做过。
【参考回答】三层各解决一个矛盾：
- 规则层解决**延迟与成本**：高频、模式固定的意图（如"查询专利号 CNxxxx"）用关键词+正则毫秒级命中，零 token 成本，且 100% 可控；
- 微调模型层解决**领域准确率**：通用大模型对专利领域的二级意图（如"检索条件补充" vs "结果反馈"）区分不好，用业务标注数据对 Qwen 做 SFT/LoRA，让模型学到领域先验；
- 大模型兜底层解决**覆盖率**：长尾、复杂表述交给 GPT-4o，保证没有请求会"无法分类"。
  单用大模型 = 又慢又贵又不稳定；单用规则 = 覆盖不了自然语言变体。三级是准确率、延迟、成本三者的帕累托组合。

**Q2：三级之间怎么决定"什么时候往下落"？置信度阈值怎么定的？**
【参考回答】规则层是精确匹配，命中即返回；微调模型输出 softmax 置信度，低于阈值（如 0.85）落到 GPT-4o。阈值是在验证集上画准确率-覆盖率曲线选的：阈值越高，模型层覆盖越少、兜底成本越高；我们选的是"模型层准确率 ≥ 规则层水平"时的最低阈值。（如果你没做过阈值实验，就说"通过在验证集上网格搜索，兼顾兜底调用比例与整体准确率确定"。）

**Q3：微调用了多少数据？怎么标注的？SFT 和 LoRA 具体怎么选的？**
【参考回答准备】必须能答：数据量级（如几千到几万条对话标注）、标注来源（线上日志人工标注 + LLM 辅助预标注再人审）、为什么用 LoRA（单卡可训、参数量小、部署时可合并权重或热插拔多任务适配器）、基座型号和参数规模（如 Qwen2-7B）、训练框架（LLaMA-Factory / peft）、评估指标（分类准确率 / F1）。**这题答不出细节 = 项目真实性存疑，务必提前把数字准备好。**

**Q4："命中率提升约 60%"是怎么算的？基线是什么？**
【意图】验证数字真实性，最常见的深挖点。
【参考回答模板】"基线是上线前单一 GPT 提示词分类的意图识别准确率约 X%，三级流水线上线后在同一测试集上到 Y%，相对提升约 60%。测试集是从线上真实请求抽样人工标注的 N 条。"——**X、Y、N 必须自己填好并背熟；如果实际没有严格测试集，就改口径为"人工抽检"并把数字说成约数。**

**Q5：结构化 Schema 的格式校验失败了怎么办？重试几次？还失败呢？**
【参考回答】用 JSON Schema 校验输出；失败则带上错误信息重新调用（最多 2 次），提示词里加入"上次输出不合法的原因"；仍失败则降级为规则层的默认意图 + 引导用户重述。上线后格式错误率近乎为零，是因为大部分流量根本走不到 LLM 层。

### 2. 意图状态机（IntentStateMachine）

**Q6：为什么用状态机而不是把历史全塞给 LLM 让它自己判断？**
【参考回答】三点：① 确定性——转移表是白盒，出了 badcase 可以精确定位到某条转移规则并修复，LLM 判断不可复现；② 成本——状态机零 token；③ 歧义本质是"同一句话在不同阶段含义不同"（如"再加一个条件"在检索前是补充、检索后可能是修改），这正是状态相关的问题，用状态机建模最直接。LLM 只在状态机无法裁决时参与。

**Q7：状态机有哪些状态？举一个具体的转移例子。**
【准备】必须能现场画出来。例如状态：初始/收集条件中/待确认/检索完成/反馈处理。例子："检索完成"状态下收到原始意图"补充条件"→ 修正为"修改检索并重新执行"，而"收集条件中"状态下同样的话 → 仅追加槽位不触发检索。

**Q8：准确率提升 25%、追问轮次减少 30% 怎么统计的？**
【参考模板】"从线上会话日志抽样 N 组多轮会话人工复核意图判断正确性，对比上线前后；追问轮次是统计每次完成一次有效检索平均需要的澄清轮数。"同样，数字口径要能自洽。

### 3. 性能优化（响应速度 40%+、Token 成本降 50%）

**Q9：异步并发具体怎么做的？和缓存分别贡献了多少？**
【参考回答】FastAPI 全链路 async；意图分类和实体抽取这类无依赖的步骤用 `asyncio.gather` 并行发起；远程 GPU 上的微调模型服务用异步 HTTP 客户端调用，不阻塞事件循环。缓存两类：① 意图结果缓存（相同/高相似 query 直接复用）；② Embedding 缓存（同一文本不重复向量化）。贡献拆分：延迟改善主要来自"规则层短路 + 并行"（LLM 往返是秒级大头），成本下降主要来自缓存和规则层拦截。**追问"40% 是 P50 还是 P99"要能答**——建议口径：平均响应时延（P50），从约 X 秒降到 Y 秒。

**Q10：缓存怎么失效？相似 query 怎么判相同？**
【参考回答】意图缓存用 query 归一化（去停用词、槽位占位化）后做键，TTL 过期；Embedding 缓存内容哈希永久有效。若做了语义相似缓存，答"embedding 余弦相似 > 阈值复用"，没做就直说精确匹配，别编。

### 4. 动态工作流引擎

**Q11：既然思想来自 LangGraph，为什么不直接用 LangGraph？**
【意图】考察造轮子的判断力。
【参考回答】当时评估过：① 业务需要深度定制的共享 State 结构和执行 Trace 格式（要对接内部监控）；② LangGraph 当时版本迭代快、API 不稳，公司代码不希望引入重依赖；③ 我们需要的子集很小（节点+条件路由+状态传递），自研 300 行内可控。核心概念确实借鉴了 LangGraph 的图编排思想。

**Q12：11 个节点 5 条路由，具体是哪些？"新增步骤 3 行代码"怎么做到的？**
【准备】能报出主要节点名：意图识别、状态机消歧、实体抽取、槽位完整性评估、追问生成、检索执行、重排、报告生成等。3 行 = 定义节点函数（装饰器注册）+ 声明入边 + 声明出边，引擎自动接管执行与 Trace。

### 5. 实体抽取与槽位记忆

**Q13：LLM 补全槽位会不会幻觉出用户没说过的条件？**
【参考回答】会，所以设计上：① LLM 只允许从对话历史文本中抽取，prompt 明确禁止推断；② 补全结果回显给用户确认后才执行检索（完整性评估节点 + 确认交互），幻觉条件在确认环节被拦截；③ 高置信槽位（正则命中的专利号）优先级高于 LLM 抽取。

## 二、项目经历：HelloAgents 多智能体框架

### 1. 整体设计

**Q14：市面上已有 LangChain/LangGraph/AutoGen，为什么自己写框架？和它们的本质区别？**
【参考回答】定位不同：这是配合教程的轻量教学/实验框架，目标是**让每个模块都能被读懂和替换**。设计差异：① Everything-as-Tools——记忆、RAG、RL、协议全部收敛为同一个 Tool 接口，Agent 侧只有"选工具、调工具"一种交互范式，没有 LangChain 那种 Chain/Agent/Memory 多套抽象；② 依赖极轻，核心零第三方框架依赖；③ 内置降级路径，没装向量库也能全流程跑通。要能坦然承认它不适合超大规模生产，适合快速原型与二次开发。

**Q15：ReAct、Reflection、Plan-and-Solve 三种范式的区别和适用场景？**
【参考回答】ReAct：思考-行动-观察循环，适合需要工具交互、信息逐步获取的任务；Reflection：生成后自我批评再修订，适合写作/代码这类质量可迭代提升的产出；Plan-and-Solve：先全局分解再逐步执行，适合步骤依赖清晰的复杂任务，缺点是计划错了会一路错，所以常配重规划。要能说出各自 prompt 结构和终止条件。

### 2. 记忆系统（必被深挖）

**Q16：四类记忆分别存什么？怎么决定一条信息进哪类？**
【参考回答】工作记忆=当前会话上下文（容量小、TTL 短）；情景记忆=带时间戳的历史事件/对话片段；语义记忆=沉淀出的事实与知识（实体、关系，入 Neo4j 图）；感知记忆=多模态原始输入的表征。写入时由 MemoryManager 按信息类型路由：对话轮次默认进工作+情景，抽取出的事实进语义。参考了认知科学的记忆分类（Atkinson-Shiffrin / Tulving），说得出这个出处是加分项。

**Q17：重要性评分、衰减、巩固具体怎么实现？**
【参考回答】重要性=写入时的启发式打分（含关键实体、用户显式强调、任务相关度）；衰减=检索得分随时间指数衰减（近期优先）；巩固=情景记忆中被反复检索/高重要性的内容定期提炼进语义记忆（类似"复习进长期记忆"）。检索时综合 相关性(向量相似)×重要性×新近度 排序——这个三因子打分与 Stanford Generative Agents 的记忆检索设计一致，可以引用。

**Q18：为什么同时要 Qdrant 和 Neo4j？**
【参考回答】职责不同：Qdrant 做**相似性检索**（情景/语义记忆的向量召回）；Neo4j 存**结构化关系**（实体-关系图，支持多跳查询，如"用户提过的项目 A 依赖哪些技术"）。向量库答"像什么"，图库答"和什么有关"。同时都做了缺失降级：没有 Qdrant 退内存向量检索，没有 Neo4j 退文档存储。

**Q19：「重复追问减少约 30%」怎么测的？**
【模板】构造多轮任务测试集（同一信息在早期轮次给出、后期需要用到），对比开/关记忆系统时 Agent 重复向用户索要已知信息的次数。**这是自测口径，主动说明是内部基准，别包装成线上数据。**

### 3. GSSC 上下文工程（结合记忆——你要求突出的重点）

**Q20：GSSC 四步具体做什么？和"把历史全拼进 prompt"比好在哪？**
【参考回答】Gather：从对话历史、记忆系统、RAG 知识库、工具结果四个源收集候选 ContextPacket；Select：按相关性得分 + 优先级 + 多样性（去冗余）筛选；Structure：按"系统指令→背景知识→相关记忆→近期对话→当前任务"模板组织；Compress：超预算时先截断低优先级包、再对长包做摘要压缩。好处：token 预算一定时，装进去的都是高价值信息——长对话下"全拼"会把关键信息挤出窗口或稀释注意力（lost in the middle），GSSC 是主动的信息配给。

**Q21：「有效信息密度提升 40%」这个指标怎么定义？**
【诚实模板】"定义为上下文中与当前任务相关的 token 占比，用一批标注任务对比 GSSC 与朴素截断两种构建方式。是内部实验口径。"——如果答不出定义，这个数字建议从简历上改成定性描述。

**Q22：Select 阶段的相关性分数怎么来的？多样性怎么做？**
【参考回答】相关性=候选包与当前 query 的 embedding 余弦相似度（无向量依赖时退 TF-IDF）；多样性=贪心去重，与已选包相似度过高的降权（类似 MMR）。

**Q23：token 计数怎么做的？不同模型分词不同怎么办？**
【参考回答】用 tiktoken 按目标模型的 encoding 计数；非 OpenAI 系模型用近似（字符数/系数）+ 预留安全余量。"上下文溢出清零"就是因为 Compress 阶段强制预算硬约束。

### 4. RAG 模块

**Q24：切块策略是什么？chunk 大小怎么选？**
【参考回答】按文档结构（标题/段落）优先切分，超长段落再按句子聚合到目标长度（如 300-500 token），带 overlap（如 50 token）防止语义断裂。chunk 太小召回碎、太大噪声多，我们按检索命中率实验选的。

**Q25：TF-IDF 降级和向量检索效果差多少？为什么选 TF-IDF 不是 BM25？**
【参考回答】降级路径牺牲语义泛化（同义改写召回差），但精确关键词场景够用，核心价值是**零依赖可运行**。（若实现确实是 TF-IDF：坦承 BM25 更好，TF-IDF 是 sklearn 现成、实现最简的选择，生产建议换 BM25/混合检索——主动说出这个权衡反而加分。）

**Q26：引用溯源怎么实现的？「可溯源率 100%」什么意思？**
【参考回答】每个 chunk 入库时带 doc_id/位置元数据，生成答案时强制模板引用来源编号，答案只允许基于检索片段生成；100% 指每条答案都能定位到源文档片段（这是机制保证，不是模型自觉）。

### 5. RL 与评测

**Q27：RL 训练具体训了什么？奖励函数怎么设计？**
【参考回答】封装 TRL 的 SFT/PPO/DPO 流程，面向工具调用场景：数据集模块构造(指令,轨迹)对；奖励函数对轨迹打分——工具选择正确性、参数合法性、最终任务完成度加权。要诚实说明规模：教学级实验（小模型/小数据验证管线通），不是大规模 RLHF。**如果被问 PPO 原理**：策略梯度 + 重要性采样 + clip 约束新旧策略比值，价值函数做 baseline 降方差；DPO 则是免 RM 直接用偏好对优化。

**Q28：BFCL、GAIA 分别测什么？你们的分数？**
【参考回答】BFCL 测函数调用（AST 匹配函数名/参数正确性），GAIA 测通用助手多步推理+工具使用。重点讲你搭的是**评测管线**（数据加载、执行、判分、报告），分数用于纵向对比迭代效果而非刷榜。「工具调用命中率提升约 60%」指接入评测驱动迭代（改进工具 description、few-shot、参数 schema）前后，框架在自建工具调用测试集上的准确率变化——口径要背熟。

**Q29：LLM-as-Judge 的可靠性问题怎么处理？**
【参考回答】已知偏差：位置偏好、长度偏好、自我偏好。缓解：评分标准 rubric 化、随机交换 A/B 位置、多次采样取多数、抽样人工校验一致性。

## 三、技能特长深挖

**Q30：LoRA 的原理？rank 怎么选？**
【参考回答】冻结原权重，为目标层注入低秩分解 ΔW=BA（r≪d），只训 A/B，参数量降几个数量级；推理可合并回原权重零开销。rank 常用 8-64，任务越复杂/数据越多可适当加大，配合 alpha 缩放。加分点：知道 QLoRA（4bit 量化基座+LoRA）进一步省显存。

**Q31：SFT、DPO、PPO(RLHF) 的关系与选择？**
【参考回答】SFT 教"会做"（模仿标注答案）；对齐阶段让模型"做得合人意"：RLHF 用奖励模型+PPO 在线优化，效果上限高但训练不稳、成本高；DPO 把偏好学习转成对比损失的离线优化，免 RM、稳定简单，是当前默认首选。我在意图分类器上用的是 SFT/LoRA，DPO 做过对齐实验。

**Q32：量化了解到什么程度？**
【参考回答】能讲 weight-only int8/int4（GPTQ、AWQ）、加载时量化（bitsandbytes）、KV cache 量化；权衡是显存/吞吐 vs 精度，4bit 对 7B 级模型一般掉点可接受。别夸大，说明是部署使用层面而非量化算法研究。

**Q33：vLLM 为什么快？**
【参考回答】核心是 PagedAttention——KV cache 分页管理消除显存碎片，配合 continuous batching（迭代级调度，请求随时插队）大幅提升吞吐。用过的场景：部署微调后的 Qwen 提供意图分类服务。

**Q34：MCP 协议是什么？解决什么问题？**
【参考回答】Model Context Protocol，Anthropic 提出的模型-工具互操作标准：工具/资源由 MCP Server 声明，任何支持 MCP 的客户端可发现并调用，解决"每个应用为每个工具写一遍胶水代码"的 M×N 问题。我在 HelloAgents 里实现了 MCP 接入，并把远端 MCP 工具自动展开成框架内的独立 Tool，让 LLM 直接按扁平 schema 调用。

## 四、教育背景 & HR 向问题

**Q35：本科大数据管理（偏管理），为什么转向大模型/Agent？基础够吗？**
【参考回答】本科已修数据结构/机器学习/深度学习并保持前 20%，毕业后通过实习完成工程化转型（拿出实习项目说话）；硕士方向（多模态情感分析）正是延续。把"跨背景"讲成"业务理解 + 技术实现的复合优势"。

**Q36：2024.06 本科毕业，2025.09 才入学，中间一年 gap 做了什么？**
【必须准备】用实习（2025.04-08）+ 考研备考/项目自学填满时间线，逻辑要顺：毕业 → 备考 → 录取后入学前实习。**这是时间线上最显眼的空档，一定提前想好说法。**

**Q37：读研期间怎么保证长期实习 6 个月以上？导师同意吗？课程怎么办？**
【必须准备】答"研一课程已修完/集中在某学期，导师支持业界实习，可保证每周 X 天到岗"。这是 HR 决定要不要你的关键问题。

**Q38：研究方向多模态情感分析，和 Agent 岗位怎么关联？**
