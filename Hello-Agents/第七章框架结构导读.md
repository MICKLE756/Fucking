## 0. 一句话总览

HelloAgents 的设计哲学是「**薄抽象、纯原生**」：不依赖 LangChain 之类的重框架，只用 OpenAI 原生 API，自底向上分成四层，每层只做一件事、对上只暴露一个稳定接口。

```
┌─────────────────────────────────────────────────────────────┐
│  第四层 · 范式与能力层  agents/*  +  tools/*                  │  ← 你平时直接用的
│   SimpleAgent / ReActAgent / ReflectionAgent /               │
│   PlanAndSolveAgent / FunctionCallAgent  +  工具系统          │
├─────────────────────────────────────────────────────────────┤
│  第三层 · 抽象层  core/agent.py                               │  ← 所有 Agent 的共同基类
│   Agent(ABC)：约定 run()，统一历史管理                        │
├─────────────────────────────────────────────────────────────┤
│  第二层 · 数据/契约层  core/message.py + core/config.py       │  ← 跨层传递的数据结构
│   Message（对话消息）/ Config（运行配置）                     │
├─────────────────────────────────────────────────────────────┤
│  第一层 · 通信层  core/llm.py + core/exceptions.py            │  ← 和大模型说话的唯一出口
│   HelloAgentsLLM：多 provider 自动识别，invoke/think/stream   │
└─────────────────────────────────────────────────────────────┘
```

读代码建议顺序：**第一层 → 第二层 → 第三层 → 第四层**（先打地基，再看楼）。下面逐层逐文件讲。

## 第一层 · 通信层（和大模型说话）

### `hello_agents/core/llm.py` — `HelloAgentsLLM`
整个框架**唯一**和 LLM 通信的地方。上层 Agent 永远不直接碰 `openai` SDK，只调这个类。

- **构造**：`HelloAgentsLLM(model=None, api_key=None, base_url=None, provider=None, temperature=0.7, ...)`
  - 「参数优先，环境变量兜底」：不传参就从 `.env` 读 `LLM_MODEL_ID / LLM_API_KEY / LLM_BASE_URL`。
  - `_auto_detect_provider()`：按环境变量自动识别厂商（`OPENAI_API_KEY`→openai、`DASHSCOPE_API_KEY`→qwen、`DEEPSEEK_API_KEY`→deepseek、`ZHIPU/GLM`→zhipu、`KIMI/MOONSHOT`→kimi、ollama/vllm/modelscope…）。支持 11 种 provider，全部走 OpenAI 兼容协议。
  - 缺 key/base_url 会抛 `HelloAgentsException`（见下）。
- **三个核心方法**（上层只用得到这几个）：
  - `invoke(messages) -> str`：一次性拿完整回复（ReActAgent 用它）。
  - `think(messages) -> Iterator[str]`：流式逐 token 产出（带「思考」语义）。
  - `stream_invoke(messages) -> Iterator[str]`：流式拿回复。
- **怎么改**：想接一个新的兼容 OpenAI 的服务，基本只要配好 `LLM_BASE_URL`/`LLM_API_KEY`，或在 `provider="custom"` 下直接传。想加自定义协议厂商，扩 `_auto_detect_provider` / `_resolve_credentials` 即可。

### `hello_agents/core/exceptions.py` — `HelloAgentsException`
框架统一异常类型。通信层、配置缺失等都抛它，方便上层用一个 `except` 兜住并给出友好提示（demo 里就是这么做的）。

## 第二层 · 数据/契约层（层与层之间传什么）

### `hello_agents/core/message.py` — `Message`
对话消息的标准结构（基于 pydantic）。

- 字段：`content`、`role`（`Literal["user","assistant","system","tool"]`，**类型即约束**，写错 role 直接报错）、`timestamp`、`metadata`。
- `to_dict()`：转成 OpenAI API 要的 `{"role", "content"}`——这是 Message 和通信层之间的桥。
- 设计意图：让「对话历史」在全框架是**同一种对象**，而不是到处用裸 dict。

### `hello_agents/core/config.py` — `Config`
运行期配置（pydantic）。

- 字段：`default_model`、`default_provider`、`temperature`、`max_tokens`、`debug`、`log_level`、`max_history_length` 等。
- `Config.from_env()`：从环境变量装配；`to_dict()` 导出。
- 设计意图：把「行为参数」和「代码逻辑」解耦，Agent 拿一个 `Config` 就知道怎么跑。

> 这一层没有任何业务逻辑，纯粹是**数据契约**：上下层都认这两个结构，于是各层可以独立演化。

## 第三层 · 抽象层（所有 Agent 的共同祖先）

### `hello_agents/core/agent.py` — `Agent(ABC)`
抽象基类，定义「一个 Agent 至少长什么样」。

- 构造：`Agent(name, llm: HelloAgentsLLM, system_prompt=None, config=None)`——**持有一个 LLM、一份配置、一段历史**。
- 抽象方法：`@abstractmethod run(input_text, **kwargs) -> str`——**每种 Agent 必须自己实现**「怎么处理一轮输入」。这是第四层各范式的差异所在。
- 通用能力（子类白拿）：`add_message()`、`get_history()`、`clear_history()`，历史用 `list[Message]` 存。
- 设计意图：把「持有 LLM + 管历史」这些**共性**收口到基类，让每种范式只需专注实现自己的 `run()`。

## 第四层 · 范式与能力层（你直接用的东西）

### 4.1 四种经典 Agent 范式（`hello_agents/agents/`）

| 文件 | 类 | 核心思想 | 关键参数 | 何时用 |
|---|---|---|---|---|
| `simple_agent.py` | `SimpleAgent` | 最朴素：system+history+user 一次问答；可选挂工具 | `enable_tool_calling`, `tool_registry` | 普通问答、基线 |
| `react_agent.py` | `ReActAgent` | **Thought→Action→Observation 循环**，靠文本协议调工具 | `tool_registry`, `max_steps`, `custom_prompt` | 需要查资料/算数等外部信息 |
| `reflection_agent.py` | `ReflectionAgent` | **生成→自我反思→改进**多轮迭代，自带 `Memory` 记录每轮 | `max_iterations`, `custom_prompts` | 追求质量的写作/代码 |
| `plan_solve_agent.py` | `PlanAndSolveAgent` | **先 Planner 拆计划，再 Executor 逐步执行**（内含两个子角色） | `planner_prompt`, `executor_prompt` | 多步骤复杂任务 |
| `function_call_agent.py` | `FunctionCallAgent` | 用模型**原生 function calling**（JSON 工具调用）而非文本协议 | `default_tool_choice`, `max_tool_iterations` | 模型支持 tools 时更稳 |
| `tool_aware_agent.py` | `ToolAwareSimpleAgent` | 继承 `SimpleAgent`，加工具调用监听回调 | `tool_call_listener` | 想观测工具调用过程 |

**精读 `ReActAgent`（最值得读的一个）**：
- `run()` 是一个 `while current_step < max_steps` 循环：
  1. 用 `prompt_template.format(tools=..., question=..., history=...)` 拼提示词（工具清单由注册表给）。
  2. `llm.invoke()` 拿模型输出。
  3. `_parse_output()` 正则抽出 `Thought:` 与 `Action:`。
  4. `Action` 是 `Finish[...]` → 收尾返回；否则 `_parse_action()` 抽 `工具名[参数]`，交注册表 `execute_tool()` 执行，把 `Observation:` 追加进历史，进入下一步。
- 这就是「**推理 + 行动**」交替：模型负责想，工具负责做，观察结果再喂回模型。读懂这段就读懂了 ReAct。

### 4.2 工具系统（`hello_agents/tools/`）

| 文件 | 内容 | 要点 |
|---|---|---|
| `base.py` | `Tool` 抽象基类 + `ToolParameter` | 每个工具实现 `run(parameters: dict) -> str`；`get_parameters()` 声明入参 |
| `registry.py` | `ToolRegistry` + 全局 `global_registry` | 两种注册：`register_tool(Tool对象)` / `register_function(name, desc, func)`；`execute_tool(name, input)` 统一执行；`get_tools_description()` 生成喂给提示词的工具清单 |
| `builtin/calculator.py` | `CalculatorTool`（name=`python_calculator`）+ 便捷函数 `calculate()` | 用 `ast` 安全求值，支持 `sqrt/sin/log` 等，无需联网 |
| `builtin/search_tool.py` | `SearchTool` + 便捷函数 `search()` | 网页搜索，需 `TAVILY_API_KEY` 或 `SERPAPI_API_KEY` |

**工具是怎么被调用的（数据流）**：
`ReActAgent.run()` → `registry.get_tools_description()` 把工具写进提示词 → 模型输出 `calculate[15*23+45]` → `_parse_action` 抽出名字和参数 → `registry.execute_tool("calculate", "15*23+45")` → 注册表里函数工具直接 `func(input)`、Tool 对象则 `tool.run({"input": input})` → 返回字符串作为 `Observation`。

## 一条请求的完整数据流（以 ReAct 调计算器为例）

```
用户: "15 * 23 + 45 = ?"
  │
  ▼  ReActAgent.run()                         （第四层）
  ├─ registry.get_tools_description()         （第四层·工具）→ 工具清单
  ├─ 拼 prompt → llm.invoke(messages)         （第一层·通信）
  │     └─ HelloAgentsLLM 走 OpenAI 兼容接口
  ├─ 模型回: "Thought: 该算一下 / Action: calculate[15*23+45]"
  ├─ _parse_action → ("calculate", "15*23+45")
  ├─ registry.execute_tool(...)               （第四层·工具）→ "390"
  ├─ history += Observation: 390              （用 Message/list 维护，第二/三层）
  ├─ 下一步模型回: "Action: Finish[结果是 390]"
  ▼
最终答案: "结果是 390"
```

把这张图和 `react_agent.py` 对着读，四层是怎么串起来的就一目了然了。

## 怎么扩展（动手改的两个最常见入口）

**① 加一个自定义工具**（最简便方式）：
```python
from hello_agents import ToolRegistry, ReActAgent, HelloAgentsLLM

def weather(city: str) -> str:
    return f"{city} 今天晴，25℃"   # 真实实现里换成调用天气 API

reg = ToolRegistry()
reg.register_function("weather", "查询某城市天气，输入城市名。", weather)
agent = ReActAgent("天气助手", HelloAgentsLLM(), tool_registry=reg)
agent.run("北京今天天气怎么样？")
```
或继承 `tools/base.py` 的 `Tool`，实现 `run(parameters)` + `get_parameters()`，用 `register_tool()` 注册（适合需要结构化入参的工具）。

**② 加一种新的 Agent 范式**：继承 `core/agent.py` 的 `Agent`，实现 `run()` 即可——基类已经帮你管好 `llm`、`config`、历史。照着 `react_agent.py` 抄一个改逻辑是最快的。

## 与教程后续章节模块的关系（vendored 包里也带了）

`learn_version` 不止第七章，还顺带带了后续章节会用到的模块，第七章阶段可以**先不看**，知道它们在哪即可：

- `hello_agents/memory/`、`memory/rag/`：第 8 章 记忆与 RAG（向量/图存储、文档管线）。
- `hello_agents/context/`：第 9 章 上下文工程（HistoryManager 等）。
- `hello_agents/protocols/`（mcp / a2a / anp）：第 10 章 智能体通信协议。
- `hello_agents/rl/`：第 11 章 强化学习训练。
- `hello_agents/evaluation/`（gaia / bfcl 等）：评测基准。

> 提醒：上游 `tools/builtin/__init__.py` 会 eager import 评估工具，所以**完整 `import hello_agents` 需要额外 `pip install huggingface_hub`**。只玩第七章核心（LLM/Message/Config/Agent/4 种 Agent/计算器·搜索）时，装 `requirements.txt` 即可。
