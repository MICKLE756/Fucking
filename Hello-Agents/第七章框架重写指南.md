## 0. 这套框架的设计哲学（先记住）

- **薄抽象、纯原生**：不用 LangChain，只用 OpenAI 官方 SDK（任何 OpenAI 兼容服务都能接）。
- **四层抽象，自底向上**：
  ```
  第四层 范式与能力层  agents/* + tools/*      ← 你平时直接用的
  第三层 抽象层        core/agent.py            ← 所有 Agent 的共同基类
  第二层 数据/契约层   core/message.py + config.py
  第一层 通信层        core/llm.py + exceptions.py  ← 和大模型说话的唯一出口
  ```
- **除了 Agent，一切皆 Tool**：Memory / RAG / RL / 协议 在原框架里都被抽象成「工具」。第七章只需掌握核心：LLM、Message、Config、Agent 基类、4~5 种 Agent、计算器/搜索两个工具。

**重写顺序 = 阅读顺序 = 第一层 → 第四层**（先打地基再盖楼）。

## 1.2 `core/llm.py` — `HelloAgentsLLM`（整个框架唯一和 LLM 通信的地方）

**职责**：上层 Agent 永远不直接碰 `openai` SDK，只调这个类的三个方法。

### 必须实现的接口
```python
class HelloAgentsLLM:
    def __init__(self, model=None, api_key=None, base_url=None,
                 provider=None, temperature=0.7, max_tokens=None,
                 timeout=None, **kwargs): ...

    def invoke(self, messages: list[dict], **kwargs) -> str: ...          # 一次性拿完整回复（ReAct 用它）
    def think(self, messages: list[dict], temperature=None) -> Iterator[str]: ...  # 流式逐 token
    def stream_invoke(self, messages: list[dict], **kwargs) -> Iterator[str]: ...  # think 的别名
```

### 实现要点（按这个思路写）
1. **「参数优先，环境变量兜底」**：`self.model = model or os.getenv("LLM_MODEL_ID")`，timeout 同理读 `LLM_TIMEOUT`（默认 60）。
2. **provider 解析**（可以先做"极简版"，跑通后再补全）：
   - 极简版：直接 `self.api_key = api_key or os.getenv("LLM_API_KEY")`、`self.base_url = base_url or os.getenv("LLM_BASE_URL")`。第七章配了 `LLM_*` 三件套就够。
   - 完整版：抄原码的 `_auto_detect_provider()`（按 `OPENAI_API_KEY/DEEPSEEK_API_KEY/...` 环境变量、key 前缀、base_url 域名自动识别 11 种厂商）+ `_resolve_credentials()`（按 provider 填默认 base_url）+ `_get_default_model()`。**建议先写极简版，理解主流程后再回来补**。
3. **校验**：`if not all([self.api_key, self.base_url]): raise HelloAgentsException("API密钥和服务地址必须被提供或在.env文件中定义。")`
4. **建客户端**：`self._client = OpenAI(api_key=..., base_url=..., timeout=...)`
5. **`invoke`**：
   ```python
   resp = self._client.chat.completions.create(
       model=self.model, messages=messages,
       temperature=kwargs.get('temperature', self.temperature),
       max_tokens=kwargs.get('max_tokens', self.max_tokens))
   return resp.choices[0].message.content
   ```
   出错包成 `HelloAgentsException`。
6. **`think`**：同样 create 但 `stream=True`，`for chunk in resp: yield chunk.choices[0].delta.content or ""`，边收边 `print`。
7. **`stream_invoke`**：`yield from self.think(messages, kwargs.get('temperature'))`。

> 对照原码 `core/llm.py`（343 行）。`invoke` 在 320 行、`think` 在 285 行。**重点理解**：所有上层只依赖 `invoke/think` 这两个稳定接口，换厂商只改构造参数。

## 2.1 `core/message.py` — `Message`

```python
"""消息系统"""
from typing import Optional, Dict, Any, Literal
from datetime import datetime
from pydantic import BaseModel

MessageRole = Literal["user", "assistant", "system", "tool"]   # 类型即约束，写错 role 直接报错

class Message(BaseModel):
    content: str
    role: MessageRole
    timestamp: datetime = None
    metadata: Optional[Dict[str, Any]] = None

    def __init__(self, content: str, role: MessageRole, **kwargs):
        super().__init__(content=content, role=role,
                         timestamp=kwargs.get('timestamp', datetime.now()),
                         metadata=kwargs.get('metadata', {}))

    def to_dict(self) -> Dict[str, Any]:        # ← Message 和通信层之间的桥
        return {"role": self.role, "content": self.content}

    def __str__(self) -> str:
        return f"[{self.role}] {self.content}"
```

## 3.1 `core/agent.py` — `Agent(ABC)`

所有 Agent 的共同基类：约定 `run()` 抽象方法 + 统一历史管理。

```python
"""Agent基类"""
from abc import ABC, abstractmethod
from typing import Optional
from .message import Message
from .llm import HelloAgentsLLM
from .config import Config

class Agent(ABC):
    def __init__(self, name: str, llm: HelloAgentsLLM,
                 system_prompt: Optional[str] = None, config: Optional[Config] = None):
        self.name = name
        self.llm = llm
        self.system_prompt = system_prompt
        self.config = config or Config()
        self._history: list[Message] = []

    @abstractmethod
    def run(self, input_text: str, **kwargs) -> str:
        """运行Agent —— 每个子类必须实现自己的 run 逻辑"""
        ...

    def add_message(self, message: Message): self._history.append(message)
    def clear_history(self): self._history.clear()
    def get_history(self) -> list[Message]: return self._history.copy()

    def __str__(self): return f"Agent(name={self.name}, provider={self.llm.provider})"
    def __repr__(self): return self.__str__()
```
> 对照原码 `core/agent.py`（46 行）。这层暂时跑不了，等第四层有了具体 Agent 才能验证。**关键认知**：基类只定义"骨架"（历史怎么存、run 必须有），不同范式靠子类的 `run()` 区分。

## 4.1 `tools/base.py` — `Tool(ABC)` + `ToolParameter`

```python
"""工具基类"""
from abc import ABC, abstractmethod
from typing import Dict, Any, List
from pydantic import BaseModel

class ToolParameter(BaseModel):
    name: str
    type: str            # "string"/"number"/"integer"/"boolean"/"array"/"object"
    description: str
    required: bool = True
    default: Any = None

class Tool(ABC):
    def __init__(self, name: str, description: str):
        self.name = name
        self.description = description

    @abstractmethod
    def run(self, parameters: Dict[str, Any]) -> str:
        """执行工具，入参是 dict，返回字符串结果"""
        ...

    @abstractmethod
    def get_parameters(self) -> List[ToolParameter]:
        """声明这个工具接受哪些参数"""
        ...

    def to_dict(self) -> Dict[str, Any]:
        return {"name": self.name, "description": self.description,
                "parameters": [p.dict() for p in self.get_parameters()]}
```

## 4.2 `tools/registry.py` — `ToolRegistry`

工具的注册、管理、执行中心。支持两种注册：Tool 对象 / 直接注册函数。

### 必须实现
```python
class ToolRegistry:
    def __init__(self):
        self._tools: dict[str, Tool] = {}
        self._functions: dict[str, dict] = {}

    def register_tool(self, tool: Tool): ...                       # 存 Tool 对象
    def register_function(self, name, description, func): ...      # 存 func(str)->str
    def get_tool(self, name) -> Tool | None: ...
    def execute_tool(self, name: str, input_text: str) -> str: ... # ★统一执行入口
    def get_tools_description(self) -> str: ...                    # ★生成喂给提示词的工具清单
    def list_tools(self) -> list[str]: ...
    def get_all_tools(self) -> list[Tool]: ...
```

### 两个关键方法的实现
```python
def execute_tool(self, name, input_text):
    if name in self._tools:
        try:    return self._tools[name].run({"input": input_text})   # Tool 对象：包成 {"input": ...}
        except Exception as e: return f"错误：执行工具 '{name}' 时发生异常: {e}"
    elif name in self._functions:
        try:    return self._functions[name]["func"](input_text)       # 函数工具：直接调
        except Exception as e: return f"错误：执行工具 '{name}' 时发生异常: {e}"
    return f"错误：未找到名为 '{name}' 的工具。"

def get_tools_description(self):
    lines = [f"- {t.name}: {t.description}" for t in self._tools.values()]
    lines += [f"- {n}: {i['description']}" for n, i in self._functions.items()]
    return "\n".join(lines) if lines else "暂无可用工具"
```

## 4.3 `tools/builtin/calculator.py` — `CalculatorTool`

用 `ast` 安全求值的纯本地计算器（无需联网，是最稳的工具调用演示）。

```python
import ast, operator, math
from typing import Dict, Any
from ..base import Tool, ToolParameter

class CalculatorTool(Tool):
    OPERATORS = {ast.Add: operator.add, ast.Sub: operator.sub, ast.Mult: operator.mul,
                 ast.Div: operator.truediv, ast.Pow: operator.pow, ast.USub: operator.neg}
    FUNCTIONS = {'abs': abs, 'round': round, 'max': max, 'min': min, 'sum': sum,
                 'sqrt': math.sqrt, 'sin': math.sin, 'cos': math.cos, 'tan': math.tan,
                 'log': math.log, 'exp': math.exp, 'pi': math.pi, 'e': math.e}

    def __init__(self):
        super().__init__(name="python_calculator",
                         description="执行数学计算。支持基本运算、数学函数等。例如：2+3*4, sqrt(16), sin(pi/2)。")

    def run(self, parameters: Dict[str, Any]) -> str:
        expr = parameters.get("input", "") or parameters.get("expression", "")
        if not expr: return "错误：计算表达式不能为空"
        try:
            return str(self._eval(ast.parse(expr, mode='eval').body))
        except Exception as e:
            return f"计算失败: {e}"

    def _eval(self, node):
        if isinstance(node, ast.Constant): return node.value
        if isinstance(node, ast.BinOp):    return self.OPERATORS[type(node.op)](self._eval(node.left), self._eval(node.right))
        if isinstance(node, ast.UnaryOp):  return self.OPERATORS[type(node.op)](self._eval(node.operand))
        if isinstance(node, ast.Call):     return self.FUNCTIONS[node.func.id](*[self._eval(a) for a in node.args])
        if isinstance(node, ast.Name):     return self.FUNCTIONS[node.id]
        raise ValueError(f"不支持的表达式: {type(node)}")

    def get_parameters(self):
        return [ToolParameter(name="input", type="string", description="要计算的数学表达式", required=True)]

# 便捷函数：注册到 registry 时用 register_function(name, desc, calculate)
def calculate(expression: str) -> str:
    return CalculatorTool().run({"input": expression})
```
> 对照原码 `tools/builtin/calculator.py`（128 行）。

# 第四层 · Agent 范式（核心收尾）

## 5.1 `agents/simple_agent.py` — `SimpleAgent`

最朴素：system + history + user 一次问答；可选挂工具（文本协议 `[TOOL_CALL:name:params]`）。

### 最小实现（先不做工具，跑通对话）
```python
from typing import Optional
from ..core.agent import Agent
from ..core.message import Message

class SimpleAgent(Agent):
    def run(self, input_text: str, **kwargs) -> str:
        messages = [{"role": "system", "content": self.system_prompt or "你是一个有用的AI助手。"}]
        for msg in self._history:
            messages.append({"role": msg.role, "content": msg.content})
        messages.append({"role": "user", "content": input_text})

        response = self.llm.invoke(messages, **kwargs)
        self.add_message(Message(input_text, "user"))
        self.add_message(Message(response, "assistant"))
        return response
```

## 5.2 `agents/react_agent.py` — `ReActAgent` ★最值得手写的一个

**这是第七章的灵魂**。Thought（想）→ Action（调工具）→ Observation（看结果）→ … → Finish（给答案）。

### 提示词模板（类顶部常量）
```python
DEFAULT_REACT_PROMPT = """你是一个具备推理和行动能力的AI助手。...

## 可用工具
{tools}

## 工作流程
Thought: 分析问题...
Action: 选择工具，格式：
- `{{tool_name}}[{{tool_input}}]`：调用工具
- `Finish[结论]`：信息足够时给出最终答案

## 当前任务
**Question:** {question}

## 执行历史
{history}

现在开始你的推理和行动："""
```
> 注意模板里 `{{tool_name}}` 是转义的花括号（因为用 `str.format`），而 `{tools}/{question}/{history}` 是真正要替换的占位符。

### run() 的循环骨架（核心！）
```python
def run(self, input_text: str, **kwargs) -> str:
    self.current_history = []
    step = 0
    while step < self.max_steps:
        step += 1
        # 1. 拼提示词（工具清单由注册表给）
        prompt = self.prompt_template.format(
            tools=self.tool_registry.get_tools_description(),
            question=input_text,
            history="\n".join(self.current_history))
        # 2. 问 LLM
        text = self.llm.invoke([{"role": "user", "content": prompt}], **kwargs)
        # 3. 抽 Thought / Action
        thought, action = self._parse_output(text)
        if not action: break
        # 4. Finish？收尾返回
        if action.startswith("Finish"):
            answer = self._parse_action_input(action)
            self.add_message(Message(input_text, "user"))
            self.add_message(Message(answer, "assistant"))
            return answer
        # 5. 否则解析 工具名[参数] 并执行，把 Observation 喂回历史
        tool_name, tool_input = self._parse_action(action)
        if not tool_name:
            self.current_history.append("Observation: 无效的Action格式")
            continue
        observation = self.tool_registry.execute_tool(tool_name, tool_input)
        self.current_history.append(f"Action: {action}")
        self.current_history.append(f"Observation: {observation}")
    # 超步数兜底
    answer = "抱歉，我无法在限定步数内完成这个任务。"
    self.add_message(Message(input_text, "user")); self.add_message(Message(answer, "assistant"))
    return answer
```

### 三个正则解析（原码 202-222 行，照抄即可）
```python
import re
def _parse_output(self, text):                    # 抽 Thought / Action
    t = re.search(r"Thought: (.*)", text)
    a = re.search(r"Action: (.*)", text)
    return (t.group(1).strip() if t else None), (a.group(1).strip() if a else None)

def _parse_action(self, action_text):             # 抽 工具名[参数]
    m = re.match(r"(\w+)\[(.*)\]", action_text)
    return (m.group(1), m.group(2)) if m else (None, None)

def _parse_action_input(self, action_text):       # 抽 Finish[这里]
    m = re.match(r"\w+\[(.*)\]", action_text)
    return m.group(1) if m else ""
```

构造函数：`__init__(self, name, llm, tool_registry=None, system_prompt=None, config=None, max_steps=5, custom_prompt=None)`；`tool_registry` 为空时 `self.tool_registry = ToolRegistry()`；`self.prompt_template = custom_prompt or DEFAULT_REACT_PROMPT`。

> **读懂这段就读懂了 ReAct**：模型负责"想"，工具负责"做"，观察结果再喂回模型，循环到 `Finish`。数据流：`run()` → `get_tools_description()` 写进提示词 → 模型输出 `calculate[15*23+45]` → `_parse_action` 抽名字和参数 → `registry.execute_tool("calculate","15*23+45")` → 返回字符串当 Observation。对照原码 `react_agent.py`（222 行）。

## 5.3 `agents/reflection_agent.py` — `ReflectionAgent`

生成 → 自我反思 → 改进，多轮迭代，自带 `Memory` 记录每轮轨迹。

## 5.4 `agents/plan_solve_agent.py` — `PlanAndSolveAgent`

先 Planner 拆计划，再 Executor 逐步执行（内含两个子角色类）。

- `Planner.plan(question)`：让 LLM 输出 ```python [...]``` 形式的步骤列表，用 `response.split("```python")[1].split("```")[0]` + `ast.literal_eval` 解析成 `list[str]`。
- `Executor.execute(question, plan)`：`for step in plan` 逐步调 LLM，把"历史步骤与结果"累积进 prompt，返回最后一步结果。
- `run()`：`plan = planner.plan(...)`；空则返回失败信息；否则 `executor.execute(...)`。
- 构造：`__init__(..., custom_prompts=None)`，`custom_prompts` 含 `"planner"`/`"executor"` 两个键。

> 对照原码 `plan_solve_agent.py`（201 行）。适合多步骤复杂任务。

## 8.2 核心层：`memory/base.py`

三个基石，全在这一个文件里：

```python
class MemoryItem(BaseModel):              # 记忆的最小单元（pydantic）
    id: str
    content: str
    memory_type: str                      # working / episodic / semantic / perceptual
    user_id: str
    timestamp: datetime
    importance: float = 0.5               # 0~1，遗忘/整合时按它排序
    metadata: Dict[str, Any] = {}

class MemoryConfig(BaseModel):            # 各类记忆的容量/路径/阈值开关
    storage_path: str = "./memory_data"
    ...

class BaseMemory(ABC):                    # 4 种记忆的共同抽象
    @abstractmethod
    def add(self, memory_item: MemoryItem) -> str: ...
    @abstractmethod
    def retrieve(self, query: str, limit: int = 5, **kwargs) -> List[MemoryItem]: ...
    @abstractmethod
    def update(self, memory_id: str, content: str, **kwargs) -> bool: ...
    @abstractmethod
    def remove(self, memory_id: str) -> bool: ...
    @abstractmethod
    def clear(self) -> None: ...
```

**重写要点**：`BaseMemory` 和第七章的 `Tool(ABC)`、`Agent(ABC)` 思路一模一样——先定抽象契约，4 种记忆各自实现。`importance` 是后面「遗忘」「整合」的排序依据，别省。

## 8.3 类型层：4 种记忆（先只做 Working）

- **WorkingMemory（工作记忆）**：短期、带容量上限的最近上下文。**极简版只做它就够跑通**。内部就是个有上限的 list，`retrieve` 按关键词重叠 + 时间倒序返回。
- **EpisodicMemory（情景记忆）**：把一次次交互打包成 `Episode`（事件序列），支持按时间/相似度回放。
- **SemanticMemory（语义记忆）**：抽取 `Entity` / `Relation` 建知识图谱，完整版落到 Neo4j。**最重最复杂（1182 行），最后做或直接降级**。
- **PerceptualMemory（感知记忆）**：把图片/音频等 `Perception` 存为多模态记忆，靠扩展名或 `modality` 参数区分。

每种都继承 `BaseMemory` 实现那 5 个抽象方法。签名统一：

```python
class WorkingMemory(BaseMemory):
    def __init__(self, config: MemoryConfig, storage_backend=None): ...
    def add(self, memory_item: MemoryItem) -> str: ...
    def retrieve(self, query: str, limit: int = 5, user_id: str = None, **kwargs) -> List[MemoryItem]: ...
```

## 8.4 核心调度：`memory/manager.py` 的 `MemoryManager`

这是记忆系统的「Agent 基类」级别的角色——统一对外，内部分发到 4 种记忆：

```python
class MemoryManager:
    def add_memory(self, content, memory_type="working", importance=0.5, ...) -> str   # 58-103
    def retrieve_memories(self, query, memory_types=None, limit=5, ...) -> List[...]    # 105-150
    def update_memory(self, memory_id, content, ...) -> bool
    def remove_memory(self, memory_id) -> bool
    def forget_memories(self, strategy="importance_based", threshold=0.1, ...)          # 194-218
    def consolidate_memories(self, from_type="working", to_type="episodic", ...)        # 220-261
    def get_memory_stats(self) -> Dict
    def clear_all_memories(self) -> None
```

**重写要点**：
- `add_memory` 里如果没显式给 `memory_type`，可以做一个 `_classify_memory_type` 自动分类（关键词/规则即可），这是「智能记忆」的关键。
- **遗忘（forget）** 三种策略：`importance_based`（低于阈值删）/ `time_based`（超 `max_age_days` 删）/ `capacity_based`（超容量删旧的）。
- **整合（consolidate）** 把高于 `importance_threshold` 的工作记忆「固化」成情景/语义记忆——模拟人类睡觉时把短期记忆转长期。

# 第九章 · 上下文工程（Context Engineering）

> 让 Agent 在长对话里**不爆窗、不烧钱、不丢缓存**。这一章不是「写个 prompt 模板」那么简单——它是一整套工程：历史压缩、Token 计数、工具输出截断、上下文装配。对照原项目 `hello_agents/context/` 下的 **4 个实现文件**（共 ~640 行）。
>
> ⚠️ 两个容易被误导的点：
> 1. `examples/chapter09_context_engineering.py` 是个 23 行的 `TODO` 占位 stub，**真正实现全在 `hello_agents/context/` 里**，别照示例写。
> 2. 本仓库当前 vendored 的 `context/` **只带了 `builder.py` 一个文件**，缺了原项目的 `history.py` / `truncator.py` / `token_counter.py`，且 `core/message.py` 还**没有 `summary` 角色和 `from_dict`**——这正是「感觉太简单」的根源。本章按**原项目的完整设计**来写，缺的部分一并补上。

## 9.0 这一章解决什么（先记住）

长对话会同时撞上三堵墙，上下文工程逐一拆解：

```
墙①  历史无限增长 → 爆上下文窗口        ⇒  HistoryManager  历史压缩（summary + 最近 N 轮）
墙②  每轮重算 token、改历史使 KV Cache 失效 ⇒  TokenCounter    缓存 + 增量计数；只追加不编辑
墙③  单个工具输出几万行塞满上下文        ⇒  ObservationTruncator  统一截断 + 落盘完整输出
———————————————————————————————————————————————————————————
装配  把「指令/状态/证据/历史」按预算拼成一段结构化上下文 ⇒ ContextBuilder（GSSC 流水线）
```

一句话：**前三个组件「省」，ContextBuilder「拼」。** 四个合起来才是完整的「上下文工程」，只写 ContextBuilder 是不够的。

## 9.2 先决条件：给 `Message` 加 `summary` 角色

`HistoryManager.compress()` 会塞进一条 `role="summary"` 的消息，序列化又要存 `metadata/timestamp`——而本仓库的 `Message` 目前是：

```python
MessageRole = Literal["user", "assistant", "system", "tool"]   # ← 缺 "summary"
def to_dict(self): return {"role": self.role, "content": self.content}  # ← 丢了 metadata/timestamp，也没有 from_dict
```

改造三处：

```python
MessageRole = Literal["user", "assistant", "system", "tool", "summary"]   # +summary

def to_dict(self) -> dict:                 # 完整序列化（会话保存要用）
    return {
        "role": self.role,
        "content": self.content,
        "timestamp": self.timestamp.isoformat() if self.timestamp else None,
        "metadata": self.metadata or {},
    }

@classmethod
def from_dict(cls, data: dict) -> "Message":
    ts = data.get("timestamp")
    return cls(
        content=data["content"], role=data["role"],
        timestamp=datetime.fromisoformat(ts) if ts else datetime.now(),
        metadata=data.get("metadata", {}),
    )
```

> 坑：`to_dict` 喂给 OpenAI API 时只认 `role/content`。如果你把它同时当「发给模型」和「存盘」两用，要么拆成 `to_api_dict()` / `to_dict()` 两个方法，要么发请求时只取这两个键。别让 `summary` 角色或多余字段直接进 API 请求。

## 9.3 `token_counter.py` — `TokenCounter`（先做，最独立）

职责：**本地预估 token，带缓存 + 降级**，让「要不要压缩」从 O(n) 变 O(1)。

```python
class TokenCounter:
    def __init__(self, model: str = "gpt-4"):
        self.model = model
        self._encoding = self._get_encoding()   # encoding_for_model → 失败退 cl100k_base → 再失败 None
        self._cache: dict[str, int] = {}        # "role:content" -> tokens

    def count_message(self, message) -> int:    # 带缓存；每条额外 +4 模拟角色标记开销
        key = f"{message.role}:{message.content}"
        if key in self._cache: return self._cache[key]
        n = self._count_text(message.content) + 4
        self._cache[key] = n
        return n

    def count_messages(self, messages) -> int:  # 累加 count_message
        ...
    def count_text(self, text) -> int: ...       # 无缓存版
    def _count_text(self, text) -> int:          # 有 encoding 用 tiktoken，否则 len(text)//4
        ...
    def clear_cache(self): ...
    def get_cache_size(self) -> int: ...
    def get_cache_stats(self) -> dict:           # {"cached_messages": n, "total_cached_tokens": sum}
        ...
```

重写要点：
- `_get_encoding` 三级降级：`encoding_for_model(model)` → `get_encoding("cl100k_base")` → `None`（无网/无包也不崩）。
- 缓存键用 `role:content`，命中直接返回；这是「只追加不编辑」能省 50-90% 重算的关键。
- `+4` 是粗略模拟每条消息角色/分隔符的固定开销，不必精确。

## 9.4 `truncator.py` — `ObservationTruncator`（工具输出统一截断）

职责：**一个工具吐 10MB 日志，不能整个塞进上下文**。统一在这里截断、并把完整输出落盘到 `tool-output/`，预览里给出文件路径。

```python
class ObservationTruncator:
    def __init__(self, max_lines=2000, max_bytes=51200,
                 truncate_direction="head", output_dir="tool-output"):
        os.makedirs(output_dir, exist_ok=True)   # 构造时就建目录
        ...

    def truncate(self, tool_name: str, output: str, metadata=None) -> dict:
        # 行数和字节都没超 → 原样返回 truncated=False
        # 超了 → 按 direction 截断 + 落盘完整输出
        return {
            "truncated": bool,
            "preview": str,                 # 截断后的预览
            "full_output_path": str | None, # 落盘路径（未截断时 None）
            "stats": {...},                 # original_lines/bytes、kept_lines/bytes、time_ms
        }

    def _truncate_lines(self, lines: list) -> list:   # head / tail / head_tail（中间塞 "...(中间省略)..."）
        ...
    def _save_full_output(self, tool_name, output, metadata) -> str:  # 写 tool_<时间戳>_<name>.json
        ...
```

重写要点：
- **判定条件是「或」**：`len(lines) > max_lines` 或 `bytes > max_bytes` 任一超标就截断。
- 三种方向：`head`（留头，适合错误/日志）、`tail`（留尾，适合实时输出）、`head_tail`（头尾都留，中间省略）。
- 截断的同时把**完整内容写 JSON 落盘**（含 tool 名、时间戳、metadata），预览里给 `full_output_path`，需要时可回查。

## 9.5 `history.py` — `HistoryManager`（历史压缩，缓存友好）

职责：**只追加、不编辑**（保 KV Cache），到达阈值就把旧历史换成一条 `summary` + 保留最近 N 轮。

```python
class HistoryManager:
    def __init__(self, min_retain_rounds=10, compression_threshold=0.8):
        self._history: list[Message] = []
        ...

    def append(self, message): ...           # 只 append
    def get_history(self) -> list: ...        # 返回副本
    def clear(self): ...

    def estimate_rounds(self) -> int:         # 一轮 = 1 条 user + 其后若干非 user
    def find_round_boundaries(self) -> list:  # 每个 user 消息的下标，如 [0,3,7,10]

    def compress(self, summary: str) -> None:
        # rounds <= min_retain_rounds → 不压
        # 否则 keep_from = boundaries[-min_retain_rounds]
        # 新历史 = [Message(summary, role="summary", metadata={"compressed_at": ...})] + self._history[keep_from:]
        ...

    def to_dict(self) -> dict:                # {"history":[msg.to_dict()...], "created_at", "rounds"}
    def load_from_dict(self, data): ...       # Message.from_dict 还原
```

重写要点：
- **轮次定义**要和 `estimate_rounds` / `find_round_boundaries` 一致：以 `role=="user"` 作为一轮的起点。压缩按「轮」对齐，不要从中间切断一轮。
- `compress` 只替换**最旧的部分**，最近 `min_retain_rounds` 轮原封不动 → 既省 token 又不破坏近端缓存。
- `summary` 用 `Message(role="summary")` 承载（所以 9.2 必须先加这个角色）。
- `to_dict/load_from_dict` 依赖 `Message.to_dict/from_dict` 的完整序列化（同样在 9.2 补）。
- `compression_threshold` 在原码里是预留字段（compress 本身按轮次判断）；真正「该不该压」可由调用方用 `TokenCounter` 算占用比和它比较。

## 9.6 `builder.py` — `ContextBuilder`（GSSC 装配流水线）

把上面省下来的材料**拼**成一段结构化上下文。GSSC = **Gather 捞 → Select 挑 → Structure 排 → Compress 压**。

### 三个结构（务必照实，别凭空加字段）

```python
@dataclass
class ContextPacket:                 # 一块上下文候选（注意：没有 source/priority 字段！）
    content: str
    timestamp: datetime = field(default_factory=datetime.now)
    metadata: dict = field(default_factory=dict)   # 用 metadata["type"] 区分来源/优先级
    token_count: int = 0             # __post_init__ 里若为 0 自动用 count_tokens 填
    relevance_score: float = 0.0

@dataclass
class ContextConfig:
    max_tokens: int = 8000
    reserve_ratio: float = 0.15      # 给输出留 15%
    min_relevance: float = 0.3       # 低于此相关性的非系统包丢弃
    enable_mmr: bool = True          # 预留多样性开关（见下方说明）
    mmr_lambda: float = 0.7
    system_prompt_template: str = ""
    enable_compression: bool = True
    def get_available_tokens(self) -> int:
        return int(self.max_tokens * (1 - self.reserve_ratio))

class ContextBuilder:
    def __init__(self, config: ContextConfig | None = None):
        self.config = config or ContextConfig()
        self._encoding = tiktoken.get_encoding("cl100k_base")
    def build(self, user_query: str,
              conversation_history: list[Message] | None = None,
              system_instructions: str | None = None,
              additional_packets: list[ContextPacket] | None = None) -> str:
        packets = self._gather(...)        # 119-155
        packets = self._select(packets, user_query)   # 157-212
        text    = self._structure(packets, user_query, system_instructions)  # 214-267
        return self._compress(text)        # 269-296
```

### 四步分别怎么写

- **`_gather`**：把候选都包成 `ContextPacket`，用 `metadata["type"]` 标来源——`instructions`（P0 系统指令）/ `task_state` / `history`（取最近 10 条，拼成 `[role] content`）。额外材料用 `additional_packets` 传入。
  > 原项目这一版**已把 MemoryTool/RAGTool 移除**（注释写明「如需使用请自行实现」）。本仓库 vendored 的 `builder.py` 反而**多接了** `memory_tool`/`rag_tool`（在 `_gather` 里 `run({"action":"search",...})` 捞 `related_memory`/`knowledge_base`）。两种都对：要带检索就给 `__init__` 加 `memory_tool=None, rag_tool=None` 两个可选参数，两个为 `None` 时只用 instructions/history，别 `NoneType` 崩。
- **`_select`**：相关性 = query 与 content 的**关键词重叠率**（`len(交集)/len(query 词)`）；新近度 = `exp(-Δ秒/3600)` 指数衰减；综合分 `0.7*相关性 + 0.3*新近度`。系统指令（`type=="instructions"`）**单独固定纳入、不参与排序与 min_relevance 过滤**；其余先按 `min_relevance` 过滤、再按综合分降序，在 `get_available_tokens()` 预算内贪心填充。
  > **诚实提醒**：`enable_mmr`/`mmr_lambda` 是配置位，但原码 `_select` **并未真正实现 MMR 去冗余**——它只做了「相关性+新近度」排序。把 MMR 当成「留好的扩展点」，别在文档里谎称已实现。
- **`_structure`**：固定模板排版，按 `metadata["type"]` 归段：

```
[Role & Policies]   ← instructions
[Task]              ← 当前 user_query
[State]             ← task_state
[Evidence]          ← related_memory / knowledge_base / retrieval / tool_result
[Context]           ← history
[Output]            ← 固定输出格式要求（结论/依据/风险/下一步）
```

- **`_compress`**：`enable_compression` 时，若整体超 `max_tokens` 就**按行累加截断**到预算内（原码是简单截断；进阶可换成用轻量 LLM 做高保真摘要）。
- **`count_tokens(text)`**（模块级函数，**不是** Builder 的方法）：`tiktoken` 的 `cl100k_base` 编码；异常时降级 `len(text)//4`。

## 9.7 ✅ 自测脚本 `t4_context.py`（离线、不连 LLM）

```python
from hello_agents.core.message import Message
from hello_agents.context.builder import ContextBuilder, ContextConfig, count_tokens
from hello_agents.context.token_counter import TokenCounter
from hello_agents.context.truncator import ObservationTruncator
from hello_agents.context.history import HistoryManager

def test_token_counter():
    c = TokenCounter()
    n1 = c.count_message(Message("你好世界", "user"))
    n2 = c.count_message(Message("你好世界", "user"))   # 第二次走缓存
    assert n1 == n2 and n1 > 0
    assert c.get_cache_size() == 1
    print("✅ TokenCounter OK", n1)

def test_truncator(tmp="tool-output"):
    t = ObservationTruncator(max_lines=3, output_dir=tmp)
    short = t.truncate("toolA", "a\nb")
    assert short["truncated"] is False and short["full_output_path"] is None
    long = t.truncate("toolB", "\n".join(str(i) for i in range(100)))
    assert long["truncated"] is True and long["full_output_path"]   # 完整输出已落盘
    print("✅ ObservationTruncator OK", long["stats"]["kept_lines"])

def test_history():
    h = HistoryManager(min_retain_rounds=2)
    for i in range(5):                       # 造 5 轮
        h.append(Message(f"问{i}", "user"))
        h.append(Message(f"答{i}", "assistant"))
    assert h.estimate_rounds() == 5
    h.compress("前几轮的摘要")
    hist = h.get_history()
    assert hist[0].role == "summary"         # 头部变成 summary（需 Message 支持该角色）
    assert h.estimate_rounds() == 2          # 只剩最近 2 轮
    print("✅ HistoryManager OK", len(hist))

def test_builder():
    b = ContextBuilder(config=ContextConfig(max_tokens=2000, reserve_ratio=0.25))
    ctx = b.build(
        user_query="帮我优化这段排序代码",
        system_instructions="你是资深 Python 工程师，回答给出复杂度分析。",
        conversation_history=[Message("上一轮：用户贴了 bubble sort", "assistant")],
    )
    assert "[Role & Policies]" in ctx and "[Task]" in ctx and "[Output]" in ctx
    assert count_tokens(ctx) <= 2000
    print("✅ ContextBuilder(GSSC) OK", count_tokens(ctx))

if __name__ == "__main__":
    test_token_counter(); test_truncator(); test_history(); test_builder()
```

## 9.8 常见坑

1. **凭空给 `ContextPacket` 加 `source`/`priority` 字段**：原码里没有，优先级/来源全靠 `metadata["type"]` 表达。照旧版指南写会和源码对不上。
2. **`build()` 参数名记错**：是 `build(user_query, conversation_history, system_instructions, additional_packets)`，不是 `build(query, instructions, task_state)`。
3. **把 `count_tokens` 当成 Builder 的方法**：它是 `builder.py` 里的**模块级函数**；`TokenCounter` 才是带缓存的类。两者别混。
4. **`enable_mmr=True` 就以为去重了**：参考实现没做 MMR，需要嵌入相似度自己补；没嵌入时退回纯关键词去重。
5. **`tiktoken` 无网下载 BPE 失败**：`_count_text`/`count_tokens` 要降级成 `len/4`，别直接抛异常。
6. **`Message.to_dict` 两用冲突**：发 OpenAI API 只认 `role/content`，存盘要 `metadata/timestamp`——别把存盘版直接塞进 API 请求（`summary` 角色 OpenAI 也不认）。
7. **历史从轮次中间切断**：压缩/截断要按 `find_round_boundaries` 的轮边界对齐，否则会留下「有答无问」的半截轮次。

# 第十章 · 智能体通信协议（MCP / A2A / ANP）

## 10.0 三个协议各管什么

| 协议 | 全称 | 解决什么 | 类比 |
|---|---|---|---|
| **MCP** | Model Context Protocol | Agent ↔ **工具/资源** 的标准接口 | 「USB 接口」——插上就能用别人的工具 |

| **ANP** | Agent Network Protocol | Agent 在**网络**里注册、发现、路由 | 「黄页 / DNS」 |

## 10.2 重写策略：优雅降级是第一原则

这一章和别处不同——`fastmcp` / `mcp` / `a2a-sdk` 都是**重量级可选依赖，没装就该静默降级**，不能让 import 直接崩。原码模式（务必照抄）：

```python
            raise HelloAgentsException("MCP 需要 fastmcp，请 pip install hello-agents[protocols]")
        ...
```

## 10.3 MCPTool：最常用，重点做

`MCPTool` 把一个 MCP 服务器包成第七章的 `Tool`，`run` 按 `action` 分发：

```python
class MCPTool(Tool):
    def __init__(self, server_command=None, server_url=None, ...):
        # 连接 MCP server（stdio 或 http/sse），自动拉取远端 tools 生成 description
        ...
    def run(self, parameters):
        action = parameters.get("action")
        # list_tools / call_tool / list_resources / read_resource / list_prompts / get_prompt
        ...
```

注意 MCP 客户端是 **async** 的，而第七章的 `Tool.run` 是同步的——原码用「在新线程里跑 event loop」(`run_in_thread` + `asyncio.run`) 把 async 桥接成 sync。重写时这个桥接别漏。

`A2ATool` / `ANPTool` 同样继承 `Tool`：A2A 负责把任务委托给远端 Agent，ANP 负责 `register_service` / `discover_service` / 路由。


## 10.5 常见坑

1. **async/sync 桥接**：MCP/A2A 底层 async，包进同步 `Tool.run` 时务必用独立线程跑 loop，否则和外层已有的 event loop 冲突报 `RuntimeError: ... already running`。
2. **`a2a-sdk` 版本漂移**：A2A 仍在早期（`a2a-sdk>=0.1.0`），接口不稳，重写时把它当「可选 + 占位实现」。
3. **ANP 是概念性实现**：`anp/implementation.py` 偏教学（内存里的服务注册表），别期待生产级网络栈。
4. **MCP server 路径/命令错**：`server_command` 要能在当前环境直接拉起，注意 cwd 和 Python 解释器。

# 第十一章 · Agentic-RL（强化学习训练）

> 训练 / 微调一个会用工具、会推理的模型。对照原码 `hello_agents/rl/`，整合层 `tools/builtin/rl_training_tool.py`。
> 示例：`examples/chapter11_RL.py`（349 行）。安装：`pip install hello-agents[rl]`（trl、transformers、torch、datasets、accelerate、peft、bitsandbytes）。

## 11.0 大图：LLM 的成长路线

```
预训练(别人做)  →  SFT(监督微调，学会"照着示范答")  →  RL(GRPO/PPO，按奖励自我优化)
```

## 11.2 配置中心：`TrainingConfig`（dataclass）

```python
@dataclass
```

## 11.3 三个训练器：统一 `.train()`

```python
class BaseTrainerWrapper:
    def __init__(self, config: TrainingConfig = None): ...
    def train(self): ...                  # 子类各自实现

class SFTTrainerWrapper(BaseTrainerWrapper):   # 监督微调（先做）
```

它们底层包的是 HuggingFace **TRL** 库的 `SFTTrainer` / `GRPOTrainer` / `PPOTrainer`，外面套 LoRA（peft）和我们的 `TrainingConfig`。**重写要点**：用 `TRL_AVAILABLE` flag + `check_trl_installation()` 在没装 trl 时给出 `get_installation_guide()` 的友好提示，而不是 ImportError 崩掉。

## 11.4 奖励函数：`rewards.py`

RL 阶段的灵魂——告诉模型「什么样的回答好」：

```python
class MathRewardFunction:                 # 数学题：答案对不对（抽取最终数字比对）
    def __call__(self, completions, **kwargs) -> List[float]: ...

create_accuracy_reward()        # 准确性奖励
create_length_penalty_reward()  # 太长扣分

```

## 11.7 常见坑

1. **没 GPU / 显存不足**：用默认 `Qwen/Qwen3-0.6B` + `use_lora=True` + 小 batch；`bitsandbytes` 量化进一步省显存（但 Windows 上 `bitsandbytes` 装起来麻烦，可不开量化）。
2. **trl 版本 API 变动大**：原码锁 `trl>=0.24.0`，TRL 的 `GRPOTrainer` 接口在多个版本间改过名，重写时以装到的版本为准。
3. **数据集联网下载失败**：`GSM8K` 走 HuggingFace；无网就用 `create_math_dataset` 的本地合成样本。
4. **奖励函数返回长度要对齐 completions**：`__call__` 必须返回和输入 batch 等长的 `List[float]`，否则 TRL 报错。

# 第十二章 · 智能体性能评估（Evaluation）

## 12.0 两个标准基准 + 数据质量评估

| 基准 | 全称 | 测什么 |

## 12.4 常见坑

1. **BFCL 依赖与主依赖冲突**：`bfcl-eval` 强制 `numpy==1.26.4`，和主项目 `numpy>=2.0` 冲突，**不能** `pip install hello-agents[evaluation-bfcl]`。按 `pyproject.toml` 注释，单开一个虚拟环境装（`uv venv .venv-bfcl` → 装 `numpy==1.26.4 bfcl-eval ...`）。
2. **GAIA 数据集要 HuggingFace 鉴权**：部分 GAIA 子集 gated，需要 `huggingface-cli login`。
3. **评估要花 LLM 调用**：`max_samples` 先设小（3~5）跑通流程，再放大。
4. **LLMJudge 偏向长答案**：用 `WinRateEvaluator` 做两两对比能缓解长度偏置。
