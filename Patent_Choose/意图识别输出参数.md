# 意图识别模块 → 检索服务 传递参数

本文档说明意图识别与交互编排模块经过多轮对话后，最终传递给成员4检索服务（`POST /retrieve/search`）的参数。

**以下所有参数均已在项目代码中实现。**

---

## 参数总览

| 参数 | 类型 | 必填 | 代码来源 | 说明 |
|------|------|------|---------|------|
| query | string | 是 | `state["user_input"]` | 用户原始输入文本 |
| session_id | string | 否 | `app.py` 自动生成 UUID | 会话追踪 ID，重置时刷新 |
| tech_field | string | 否 | `state["tech_domain"]` | 技术领域 |
| core_problem | string | 否 | `state["core_problem"]` | 核心技术问题 |
| constraints | object | 否 | `state["constraints"]` | 结构化约束条件 |
| filters | object | 否 | `state["filters"]` | 硬过滤条件（从 constraints 自动拆分） |
| top_k | integer | 否 | `_search_patents(top_k=10)` | 返回结果数量，默认 10 |

---

## 输出参数结构

```json
{
  "query": "耐高温400度以上、用于餐饮厨具的不粘锅涂层材料",
  "session_id": "a3f1b2c8",
  "tech_field": "涂层",
  "core_problem": "耐高温不粘锅涂层",
  "constraints": {
    "temperature_resistance": "400度以上",
    "application": "餐饮厨具",
    "environmental_friendly": true
  },
  "filters": {
    "tech_field": "涂层",
    "publish_year_from": 2024,
    "application_scene": "餐饮厨具"
  },
  "top_k": 10
}
```

## 各字段填充规则

### query

| 项目 | 说明 |
|------|------|
| 代码位置 | `workflow_nodes.py` → `state["user_input"]` |
| 来源 | 用户原始输入文本，直接透传 |
| 必填 | 是 |

### session_id

| 项目 | 说明 |
|------|------|
| 代码位置 | `app.py` → `uuid.uuid4()[:8]` |
| 来源 | 服务启动或会话重置时自动生成 |
| 用途 | 日志追踪，不影响检索结果 |

### tech_field

| 项目 | 说明 |
|------|------|
| 代码位置 | `entity_extractor_rule_base.py` → `_extract_tech_domain()` |
| 提取方式 | 关键词匹配（涂层、汽车、电池、传感器、智能制造等 28 个关键词） |
| 补充方式 | 规则未命中时由 LLM 补充 |
| 示例 | "不粘锅涂层材料" → `"涂层、不粘锅"` |

### core_problem

| 项目 | 说明 |
|------|------|
| 代码位置 | `entity_extractor_rule_base.py` → `_extract_core_problem()` |
| 提取方式 | 正则匹配 `(想找\|查找\|检索)(.*?)(专利\|技术)` |
| 补充方式 | 规则未命中时由 LLM 补充 |
| 示例 | "我想找一种耐高温的不粘锅涂层材料相关的专利" → `"耐高温的不粘锅涂层材料相关"` |

### constraints

实体抽取器 `_extract_constraints()` 通过正则 + 关键词提取：

| 键名 | 提取方式 | 代码中的正则/关键词 | 示例 |
|------|---------|-------------------|------|
| temperature_resistance | 正则 `(\d+)\s*[°℃度]` | "400度以上" → `"400度以上"` |
| application | 正则 `(用于\|适用于)(.*?)[\s，]` + 关键词（餐饮、厨具、工业等） | "用于餐饮厨具" → `"餐饮厨具"` |
| material_type | 关键词（陶瓷、氟聚合物、纳米、金属等） | "陶瓷基" → `["陶瓷"]` |
| environmental_friendly | 关键词（环保、PFOA、无毒等） | "环保" → `true` |
| PFOA_free | 关键词（PFOA） | "不含PFOA" → `true` |
| legal_status | 关键词（已授权、有效等） | "已授权" → `"已授权"` |
| time_range | 正则 `(近\|最近)(\d+)(年)` / `(\d{4})[-~到至](\d{4})` | "最近两年" → `"最近2年"` |
| patent_id | 正则 `CN\d{9,13}[\.\-]?\d?` | 内部传递，不发给检索服务 |

### filters

由 `_build_filters()` 从 constraints + tech_domain 自动拆分，**不需要额外抽取**：

| 字段 | 映射逻辑 | 代码位置 | 示例 |
|------|---------|---------|------|
| tech_field | 直接取 `state["tech_domain"]` | `workflow_nodes.py:245` | `"涂层"` |
| publish_year_from | 从 `constraints["time_range"]` 解析年份 | `workflow_nodes.py:249-264` | "最近2年" → `2024` |
| application_scene | 直接取 `constraints["application"]` | `workflow_nodes.py:267-269` | `"餐饮厨具"` |

**filters 与 constraints 的区别**：
- `constraints` → 拼进 query 做**语义匹配**（模糊）
- `filters` → 检索服务做**字段级硬过滤**（精确，不符合直接丢弃）

---

## 触发条件

只有满足以下条件时，才会构造请求发送给检索服务：

1. **意图确认**：一级意图 = `search`，二级意图 = `专利检索`
2. **用户确认**：经过追问补全后，用户明确回复"确认"/"继续"/"好的"
3. **最低信息**：`query` 非空（其余字段可选）

---

## 完整流程示例

```
用户: "我想找一种耐高温的不粘锅涂层材料相关的专利"
     ↓ 意图识别: search/专利检索
     ↓ 实体抽取: tech_field="涂层、不粘锅", core_problem="耐高温的不粘锅涂层材料相关"
     ↓ 完整性评估: 缺少约束条件 → 追问

系统: "请问您对涂层的耐温性能有具体要求吗？"

用户: "需要耐受400度以上，用于餐饮厨具，最近两年的"
     ↓ 实体抽取补充:
     ↓   constraints = {temperature_resistance:"400度以上", application:"餐饮厨具", time_range:"最近2年"}
     ↓   filters 自动生成 = {tech_field:"涂层、不粘锅", publish_year_from:2024, application_scene:"餐饮厨具"}
     ↓ 完整性评估: 通过 → 确认

系统: "确认检索条件：涂层领域、耐温400℃以上、餐饮厨具场景。是否继续？"

用户: "确认"
     ↓ 构造请求 → POST /retrieve/search
```

```json
  "core_problem": "耐高温的不粘锅涂层材料相关",

    "temperature_resistance": "400度以上",
    "application": "餐饮厨具",
    "time_range": "最近2年"
  },

    "tech_field": "涂层、不粘锅",
    "publish_year_from": 2024,
    "application_scene": "餐饮厨具"
  },
  "top_k": 10
}
```
