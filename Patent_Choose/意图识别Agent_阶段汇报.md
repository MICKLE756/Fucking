# 意图识别 Agent 阶段汇报

> 所属项目：基于动态工作流与状态机的 Agent 交互式专利检索系统（`Patent_Choose`）
> 汇报时间：2026-06-11
> 更新：2026-06-23（新增第 9 节「意图识别迭代改进记录」：5 项 Bug 修复 + 离线评测脚本）

---

## 1、负责模块名称

**意图识别 Agent**

负责将用户的自然语言输入解析为结构化意图（`{一级意图: [二级意图]}`），作为整个动态工作流的入口节点（`intent_recognize`），并通过上下文状态机对意图进行消歧，驱动后续的实体抽取、完整性评估、工具执行等节点。

意图体系（5 个一级意图 / 14 个二级意图）：

| 一级意图 | 二级意图 |
|---------|---------|
| search | 专利检索、专利详情查询 |
| analysis | SWOT分析、技术对比分析、风险评估、价值评估 |
| operation | 专利聚束组合、专利收藏、导出报告 |
| feedback | 结果不满意、修改条件、换方向 |
| chitchat | 闲聊、无关输入 |

## 2、当前已完成内容

意图识别采用 **三级 Fallback 流水线 + 状态机消歧** 的架构，目前主体链路已打通并可运行：

**代码 / 接口**
- `intent_recognize_rule_base.py`（规则层，约 200 行）：关键词 + 正则的二级意图识别，覆盖全部 5 个一级意图，延迟 < 1ms，命中高频场景。
- `intent_recognize_model_base.py`（远程微调模型层，约 100 行）：基于 OpenAI 兼容 API 的 HTTP 客户端，封装 prompt 构造、请求发送、JSON 解析、一级意图合法性校验；具备完整容错降级（超时 / 连接失败 / HTTP 5xx / 格式异常 → 自动降级）。
- `intent_state_machine.py`（意图状态机，约 120 行）：5 状态（idle / need_gathering / confirming / result_presented / analyzing）+ 完整状态转移表，基于「(当前状态, 原始意图) → (新状态, 修正意图)」做上下文消歧，并与工作流 `phase` 双向同步。
- 三级 Fallback 已在 `workflow_nodes.py` 的 `_node_intent_recognize` 节点中集成：规则 → 远程微调模型 → OpenAI 兜底 → 状态机修正，统一输出 `{level1: [level2]}` 格式。
- FastAPI Web 服务（`app.py`）已提供 `/chat`、`/reset` 接口，前端 `templates/index.html` 带实时状态面板，可直观展示每轮的意图识别结果与执行 Trace。

**数据**
- 微调训练数据集已构建：`patent_intent_train.json`（140 条）+ `patent_intent_train_v2.json`（107 条），共约 250 条，采用 `instruction / input / output` 指令微调格式，输出为标准意图 JSON。
- 目标微调模型：`Qwen2.5-8B-sft-all`（部署于 Linux GPU 服务器，OpenAI 兼容接口）。

**文档 / 测试**
- README 已完整记录意图体系、三级 Fallback 架构、状态机转移表与消歧示例、远程模型通信协议。
- `test_remote_model.py`：远程微调模型连通性与分类效果的冒烟测试脚本（含规则命中 / 不命中两类用例）。

**本地验证结果（规则层 + 状态机，无需 GPU / API Key）**

| 输入 | 识别结果 |
|------|---------|
| 我想找一种耐高温的不粘锅涂层专利 | `{search: [专利检索]}` |
| 帮我查一下 CN202310001234.5 这个专利详情 | `{search: [专利检索, 专利详情查询]}` |
| 对这个专利做 SWOT 分析 | `{analysis: [SWOT分析]}` |
| 这些结果太宽泛了 | `{feedback: [结果不满意]}` |
| 你好 | `{chitchat: [闲聊]}` |

状态机消歧验证（同一句「还需要环保 / 修改条件」在不同状态下）：
- `need_gathering` 状态：`feedback/修改条件` → 修正为 `search/专利检索`（实际是补充检索条件）✅
- `idle` 状态：`feedback/修改条件` → 修正为 `chitchat/闲聊`（无上下文，不可能是反馈）✅

## 5、遇到的问题或风险

| 类型 | 问题 / 风险 | 说明 |
|------|------------|------|
| 数据 | 微调数据量不足 | 当前约 250 条，距目标（每二级意图 200–500 条、合计 2000–5000 条）差距较大，直接影响微调模型精度。 |
| 模型 | 远程模型尚未联调 | `REMOTE_MODEL_URL` 未配置，三级 Fallback 实际退化为「规则 + OpenAI 兜底」，微调模型价值未验证。 |
| 接口 | 三层输出格式一致性 | 规则 / 远程模型 / OpenAI 三层需严格统一为 `{level1: [level2]}`，任一层格式漂移会导致下游节点解析出错（已加每层校验，仍需联调验证）。 |
| 技术 | 状态机与 phase 同步 | 状态机依赖每轮显式 `phase → state` 同步，若工作流新增 phase 未同步映射，消歧会失效。 |
| 模型效果 | feedback / chitchat 易混淆 | 模糊短输入（如「换一个」「还需要环保」）单句识别歧义大，强依赖状态机修正，状态判断错误会放大误判。 |
| 联调 | 实体抽取耦合 | 意图决定后续抽取的实体类型，意图误判会直接导致抽取目标错误，需与实体抽取模块联调验证边界。 |
| 时间 | GPU 资源 / 部署排期 | 微调与部署依赖 GPU 服务器可用性，存在排期与服务器可达性风险。 |

## 9、意图识别迭代改进记录（2026-06-23）

本轮针对意图识别环节做了一次代码审查，定位并修复了 5 个真实 Bug，并新增了一个**离线评测脚本**把规则层效果量化出来（对应第 7 节下一步计划第 2 条已提前落地一部分）。

### 9.1 修复的 5 个 Bug

| # | 文件 | 问题 | 修复 |
|---|------|------|------|
| 1 | `intent_recognize_rule_base.py` | **多意图碰撞时优先级靠字典定义顺序随机决定**：规则层把所有命中的一级意图都返回，但下游只取第一个 key，而 key 顺序就是关键词定义顺序（search→analysis→operation→feedback→chitchat），并非按置信度。 | 新增 `score()` / `top_intent()`：正则命中权重 `4.0`、关键词权重 `1 + 0.4×长度`（越长越具体），一级意图取其下二级意图最高分；`__call__` 改为**按分数降序**返回，下游取第一个即真正最可能意图。 |
| 2 | `workflow_nodes.py` | **规则层「一命中就 return」**，子串误命中会挡住更准的远程模型 / OpenAI 兜底，且无任何置信度判断。 | 加入**置信度 / 分差门控**：`score < RULE_MIN_SCORE(2.0)` 或 `Top1−Top2 < RULE_MIN_MARGIN(1.0)`（模糊）时主动降级到模型 / LLM，避免单个误命中一票否决。 |
| 3 | `workflow_nodes.py` | **确认环节否定词不全**：deny 词表只有 `不对/不是/不行/修改/错/换`，导致「好的，不过我想**改**一下温度」被误判为确认并直接执行旧条件。 | 扩充 deny 词表（`改/调整/换/重新/不过/但/先不/另外/...`），任何修改诉求都不再触发「假确认」。 |
| 4 | `intent_state_machine.py` | **状态机修正时丢弃二级意图**：修正一级意图时直接用默认二级意图覆盖，丢失已识别信息。 | 改为优先保留：原二级意图在新一级意图下合法则沿用，否则才回退默认。 |
| 5 | `intent_recognize_model_base.py` | **远程模型未强制 JSON 输出**（README 协议里写了 `response_format` 但代码没带），且解析不容错，模型一旦带 ```json 围栏或多余文字就解析失败→降级。 | payload 补上 `response_format={"type":"json_object"}`，并新增 `_loads()` 容忍围栏 / 前后多余文本（抠首尾大括号再解析）。 |

**修复效果示例**（Bug #1）：

| 输入 | 修复前 | 修复后 |
|------|--------|--------|
| 这些结果太宽泛了，我只想看已授权的专利 | `search`（被字典顺序强判，feedback 信号丢失） | `feedback` ✅ |

### 9.2 新增：离线评测脚本 `eval_intent.py`

零外部依赖、零 API Key，在 247 条带标签样本（`patent_intent_train.json` + `_v2.json`）上评测**规则层**，输出：整体一级意图准确率 / 覆盖率 / 覆盖样本准确率 / 二级意图准确率 / **各意图 P-R-F1** / **混淆矩阵** / 误判样例。评测口径与线上门控阈值一致（模糊则记为「弃判 abstain」，对应线上降级到模型/LLM）。

**评测结果（默认阈值 `min_score=2.0, min_margin=1.0`）：**

| 指标 | 值 |
|------|----|
| 一级意图准确率（全部样本） | 37.7%（93/247） |
| 覆盖率（规则层敢判占比） | 38.9%（96/247） |
| **覆盖样本上的准确率** | **96.9%（93/96）** |
| 二级意图准确率（一级判对内） | 96.8%（90/93） |

**各意图 P/R/F1：**

| 意图 | Precision | Recall | F1 | 样本数 |
|------|-----------|--------|----|--------|
| search | 0.895 | 0.425 | 0.576 | 40 |
| analysis | 1.000 | 0.257 | 0.409 | 70 |
| operation | 0.955 | 0.467 | 0.627 | 45 |
| feedback | 1.000 | 0.456 | 0.627 | 57 |
| chitchat | 1.000 | 0.314 | 0.478 | 35 |
| **macro F1** | | | **0.543** | |

**结论**：规则层是**高精度、低召回的「快速门控」**——敢判时准确率 96.9%，其余 61% 主动弃判交给微调模型 / OpenAI 兜底，正好印证三级 Fallback 架构的合理性。阈值可调：放宽到 `min_score=1.0, min_margin=0.0` 时，覆盖率升到 50.2%、覆盖准确率仍有 95.2%。

该脚本可直接用于后续「接入微调模型」「加语义检索层」前后的**量化对比**，是简历里「用数字说话」的关键素材。

**运行方式：**
```bash
cd Patent_Choose/src
python eval_intent.py                          # 评测 + 混淆矩阵
python eval_intent.py --min-score 1.0 --min-margin 0.0   # 放宽阈值看覆盖率/精度权衡
```
