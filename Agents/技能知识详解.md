# 第一部分：Hello-Agents 基础知识点详解

## 1. 智能体基础（第 1~3 章）

### 1.1 什么是智能体（Agent）
**定义**：Agent = 能**感知环境、自主决策、采取行动**以达成目标的系统。和「调用一次 LLM 拿到回答」不同，Agent 强调**多步、自主、带反馈的循环**。

一个最朴素的 Agent 循环：
```
感知(Perception) → 思考/决策(Reasoning) → 行动(Action) → 观察结果(Observation) → 再思考 ...
```

**类比**：普通 LLM 调用像「问一句答一句」的问答机；Agent 像一个**实习生**——你给它目标，它会自己查资料、动手、看结果、再调整，直到完成。

### 1.2 两大流派（务必理解区别，面试常问）
- **软件工程类 Agent**（Dify / Coze / n8n）：本质是**流程驱动**，人把流程画好，LLM 只是流程里的一个「处理节点」。优点是可控、好落地；缺点是不够灵活。
- **AI 原生 Agent**（AI-Native）：由 **AI 自己驱动**决策流程，LLM 决定下一步做什么。优点是灵活、能应对开放任务；缺点是可控性、可靠性更难。

> Hello-Agents 主攻的是后者。一句话区分：**「流程是人写死的，还是 AI 当场决定的」**。

### 1.3 智能体类型
- **简单反射型**：if 条件 → 动作（无记忆）。
- **基于模型型**：维护一个对世界的内部模型。
- **目标驱动型 / 效用驱动型**：为达成目标、最大化某个效用而规划。
- **学习型**：能从经验中改进（对应后面的「自进化」）。

### 1.4 LLM 基础（第 3 章，Agent 的「大脑」）
- **Transformer / 注意力（attention）**：模型靠注意力机制衡量「每个词该关注哪些词」。
- **Token / 上下文窗口**：模型一次能看的文本有上限（如 128k tokens）。**例子**：一篇超长文档塞不进上下文，就需要 RAG（见第 6 节）。
- **采样参数**：`temperature`（越高越随机/有创意，越低越确定）、`top_p` 等。**例子**：写代码用低 temperature 保稳定，头脑风暴用高 temperature。
- **LLM 的局限**：会**幻觉**（编造事实）、知识有**截止日期**、不会算精确数学、没有真正的长期记忆 —— 这正是 Agent 要用「工具 + 记忆 + 检索」来弥补的原因。

## 2. 智能体经典范式（第 4 章）★核心

这是 Agent 的「心法」，建议**不依赖框架手写一遍**。

### 2.1 ReAct（Reasoning + Acting）
让模型**交替输出「思考」和「行动」**，行动调用工具，把工具结果作为「观察」喂回去，循环到得出答案。

**结构**：
```
Thought: 我需要先查一下今天的汇率
Action: search("USD to CNY today")
Observation: 1 USD = 7.16 CNY
Thought: 现在可以算了
Action: calculator("100 * 7.16")
Observation: 716
Thought: 得到答案了
Final Answer: 100 美元约等于 716 元人民币
```

**最小伪代码**（不用任何框架，理解 loop 本质）：
```python
def react_agent(question, tools, llm, max_steps=5):
    history = f"Question: {question}\n"
    for _ in range(max_steps):
        out = llm(REACT_PROMPT + history)      # 让模型产出 Thought + Action
        if "Final Answer:" in out:
            return out.split("Final Answer:")[-1].strip()
        action, arg = parse_action(out)        # 解析出要调哪个工具、参数是什么
        obs = tools[action](arg)               # 真正执行工具
        history += out + f"\nObservation: {obs}\n"
    return "未能在步数内完成"
```
**适用**：需要查外部信息、调工具的任务。**失败模式**：会陷入循环、解析 Action 失败、工具选错。

### 2.2 Plan-and-Solve（先规划，再执行）
先让模型**把任务拆成步骤清单**，再逐步执行。适合复杂、多步任务。

**例子**：任务「写一份某公司调研报告」→
```
Plan:
1. 收集公司基本信息
2. 查最近财报/新闻
3. 分析竞争对手
4. 汇总成报告
然后逐条执行每一步。
```
对比 ReAct：ReAct 是「走一步看一步」，Plan-and-Solve 是「先列计划再走」。复杂任务用后者更不容易跑偏。

### 2.3 Reflection（反思 / 自我批评）
让模型**对自己的输出打分、找毛病、再改进**，形成「生成 → 批评 → 修订」的闭环。

**例子（写代码）**：
```
生成: 写一个排序函数
反思: 这个函数没处理空列表，且没写测试
修订: 补上边界处理 + 单元测试
```
**Reflexion** 是其进阶：把反思结果存进记忆，下次任务时带上，越做越好。

> 三者关系：**ReAct 解决「会用工具」，Plan-and-Solve 解决「会规划」，Reflection 解决「会自我纠错」**。真实 Agent 常把它们组合起来。

## 3. 低代码平台（第 5 章）

不写代码、靠拖拽搭 Agent / 工作流的平台。

- **Coze（扣子）**：字节出品，搭聊天机器人/工作流，接插件方便，适合快速做 demo。
- **Dify**：开源 LLMOps 平台，可视化编排 + RAG + API 发布，企业用得多。
- **n8n**：通用工作流自动化（不止 AI），强在连接各种外部系统（数据库、邮件、API）。

**什么时候用低代码 vs 写代码？**
- 流程固定、要快速上线、给非技术同事用 → **低代码**。
- 逻辑复杂、需要精细控制、要做研究/创新 → **写代码**。

**例子**：给运营做一个「自动回复客户常见问题」的机器人 → Dify 一天能搭好；要做一个会自我进化的研究 Agent → 必须写代码。

## 4. 框架开发实践（第 6 章）

主流代码框架，理解各自定位：

- **AutoGen（微软）**：擅长**多 Agent 对话协作**，把多个角色 Agent 凑成一个「会议」。
- **AgentScope（阿里）**：面向多 Agent 应用，强调消息传递与分布式。
- **LangGraph（LangChain）**：用**图（状态机）** 编排 Agent 流程 —— 节点是步骤，边是流转，支持**分支、循环、人类介入**。工业界主流，强烈建议吃透。

**LangGraph 核心思想例子**：把一个「客服 Agent」画成图：
```
[接收问题] → [判断意图] --技术问题--> [查知识库] → [回答]
                         --退款--> [转人工]
                         --闲聊--> [直接回答]
[回答] → (用户不满意?) --是--> 回到[判断意图]   # 这就是循环
```
LangGraph 让你显式地把这种**带分支和循环的流程**写出来，比纯 prompt 更可控、可调试。

## 5. 自研框架（第 7 章）

从 0 造一个 Agent 框架，目的是**看穿所有框架的本质**。核心抽象通常是四层：

| 抽象 | 职责 | 类比 |
|---|---|---|
| **LLM** | 封装大模型调用（输入 prompt → 输出文本） | 大脑 |
| **Tool** | 可被调用的能力（搜索、计算、读文件…） | 手 |
| **Memory** | 存取历史/知识 | 记忆 |
| **Agent** | 把上面三者串成决策循环 | 人本身 |

**极简实现骨架**：
```python
class Tool:
    name: str
    def run(self, arg: str) -> str: ...

class Memory:
    def add(self, item): ...
    def get(self) -> list: ...

class Agent:
    def __init__(self, llm, tools: dict, memory: Memory):
        self.llm, self.tools, self.memory = llm, tools, memory
    def step(self, user_input):
        ctx = self.memory.get()
        decision = self.llm(build_prompt(ctx, user_input, self.tools))
        if decision.is_tool_call:
            result = self.tools[decision.tool].run(decision.arg)
            self.memory.add(result)
            return self.step(user_input)   # 带着工具结果继续
        return decision.answer
```
**收获**：理解了这套抽象，再看 LangChain/AutoGen 就是「同一套东西的不同封装」。

## 6. 记忆与检索 / RAG（第 8 章）★高频

### 6.1 为什么需要 RAG
LLM 知识有截止日期、装不下你的私有文档、还会幻觉。**RAG（检索增强生成）** = 先从你的资料里**检索**相关片段，再把它们塞进 prompt 让 LLM **基于事实**回答。

**流程**：
```
文档 → 切块(chunking) → 向量化(embedding) → 存进向量库
提问 → 把问题向量化 → 检索最相似的若干块 → 拼进 prompt → LLM 基于这些块回答
```

**例子**：做「公司内部制度问答」。员工问「年假几天？」→ 系统检索到员工手册里「年假」那一段 → LLM 基于该段回答，并能附出处，避免乱编。

### 6.2 关键概念
- **Chunking（切块）**：文档切多大？太大噪声多、太小丢上下文。常见按段落/标题切，带重叠。
- **Embedding（向量化）**：把文本变成向量，语义相近的向量距离近。
- **向量检索**：用余弦相似度找最近的块。
- **记忆类型**：
  - **短期/工作记忆**：当前对话上下文。
  - **长期记忆**：跨会话保存（向量库或数据库），让 Agent「记得你」。

**最小 RAG 伪代码**：
```python
chunks = split(document)                      # 1. 切块
index = [(c, embed(c)) for c in chunks]       # 2. 建索引
def answer(q):
    qv = embed(q)
    top = sorted(index, key=lambda x: cos(qv, x[1]), reverse=True)[:3]  # 3. 检索
    context = "\n".join(c for c, _ in top)
    return llm(f"基于以下资料回答：\n{context}\n\n问题：{q}")           # 4. 生成
```

## 7. 上下文工程（第 9 章）

**上下文工程** = 在有限的上下文窗口里，**放对的信息**，让 Agent 持续、连贯地工作。它比单纯「写好 prompt」更系统。

要解决的问题：
- 上下文有上限，长对话/长任务会**爆窗口**。
- 放太多无关信息会**稀释注意力**、增加成本、引发幻觉。

常见手段（配例子）：
- **裁剪/摘要**：对话太长时，把早期内容压成摘要。例：聊到第 50 轮时，把前 40 轮压成「用户在做 X，已确定 A/B/C」。
- **选择性检索**：只在需要时把相关记忆/文档拉进来，而不是全塞。
- **结构化组织**：用固定模板组织上下文（系统指令 / 工具说明 / 相关记忆 / 当前任务）。
- **成本意识**：每个 token 都要钱、都占窗口，要「精打细算」。

> 一句话：**上下文是 Agent 的稀缺资源，工程化地管理它，是从「能跑」到「跑得好」的关键。**

## 8. 智能体通信协议（第 10 章）

当 Agent 要和工具、其他 Agent 互通时，需要标准协议。

- **MCP（Model Context Protocol）**：Anthropic 提出，**已成事实标准**。统一「Agent 如何接入外部工具/数据源」。类比：**Agent 世界的 USB 接口** —— 任何工具实现 MCP server，任何 Agent 作为 client 都能即插即用。
- **A2A（Agent-to-Agent）**：Agent 之间互相通信/协作的协议。
- **ANP（Agent Network Protocol）**：面向 Agent 网络的协议。

**MCP 例子**：你写一个「查公司数据库」的 MCP server，暴露 `query_db` 工具；之后 Claude、Cursor、你自己的 Agent 都能直接调用它，无需各自重写对接代码。

> 学习建议：**亲手写一个 MCP server + client**，是 2025-2026 简历的明确加分项。

## 9. Agentic-RL（第 11 章）★进阶

用强化学习**训练**一个更会做 Agent 任务的 LLM（不只是 prompt 它）。

- **SFT（监督微调）**：用「问题→正确步骤/答案」的示范数据教模型模仿。**例子**：收集大量优秀的 ReAct 轨迹，让模型学会这种思考-行动格式。
- **RL / GRPO**：让模型自己尝试，根据**奖励**调整。**GRPO** 是一种高效的策略优化方法（DeepSeek 等用过）。
- **可验证奖励（verifiable reward）**：关键思想 —— 任务结果能被自动判对错（如数学答案、代码能否通过测试），就能用它当奖励，规模化训练。**例子**：训一个解题 Agent，答案对=+1、错=0，让模型自己探索更好的解法。

**为什么用 RL 训 Agent**：很多 Agent 能力（长链推理、工具使用策略）靠模仿学不全，需要让模型在「试错 + 奖励」中自己摸索出更优策略。

## 10. 智能体性能评估（第 12 章）★被低估但极重要

「**会做评估**」是实习简历里最稀缺的能力之一。

- **核心指标**：任务成功率、步数/轮数、token 成本、延迟、工具调用正确率。
- **基准测试（benchmark）**：用公开数据集横向比较（如 SWE-bench 测代码 Agent）。
- **LLM-as-judge**：用一个强模型当「裁判」给输出打分，适合没有标准答案的开放任务。
- **构建测试集**：自己整理一批「输入 + 期望」的样例，回归测试每次改动。

**例子**：你给 RAG 系统做了优化，怎么证明变好了？→ 准备 50 个问答对，跑改动前后，比较「答对率从 72% → 85%、平均成本下降 20%」。**有数字，简历才有说服力。**

# 第二部分：需要拓展 / 重要知识点详解

## A. 原理深挖（读一手论文，能讲透）

- **范式论文**：
  - **ReAct**（推理+行动交替）、**Reflexion**（带记忆的反思）、**Toolformer**（模型自学何时调工具）、**Tree-of-Thoughts**（把思考展开成树，多路探索再择优）。
  - 学法：每篇都说清「解决什么问题、怎么做、适用场景、失败模式」。
- **Function Calling / Tool Use**：现代 Agent 调工具的工业标准方式 —— 模型按 **JSON Schema** 输出结构化的工具调用。
  ```json
  // 模型输出（而非自由文本），程序好解析、好执行
  {"tool": "get_weather", "arguments": {"city": "Beijing", "date": "2026-06-23"}}
  ```
  难点：**可靠性** —— 工具选对、参数填对、输出严格合 schema。
- **Planning**：任务分解、Plan-and-Execute、**ReWOO**（先一次性规划好所有工具调用，减少反复调用 LLM 省成本）。

## B. RAG / 记忆（进阶）

- **Chunking 策略**：固定长度 vs 按语义/标题切 vs 重叠切。**例子**：法律合同按条款切，比按固定字数切检索更准。
- **Embedding 选型**：不同模型效果/成本不同，中文场景要选支持中文好的。
- **Rerank（重排序）**：先用向量召回 50 条，再用一个更强的 rerank 模型精排出最相关的 5 条。**例子**：召回阶段图快，rerank 阶段图准，两段式提升准确率。
- **混合检索**：**BM25（关键词）+ 向量（语义）** 结合，兼顾「精确匹配」和「语义相近」。**例子**：查「型号 X-200 的参数」，关键词检索能精确命中型号，语义检索能理解「参数」。
- **GraphRAG**：把知识构建成**图**（实体+关系）再检索，擅长回答需要跨多个文档串联的复杂问题。
- **评估检索质量**：召回率、命中率、MRR；学会分析「为什么这次没检索到」并优化。

## C. 工程化（区分「会调 API」和「能上线」）★实习最看重

- **可观测性（Observability）**：用 **Langfuse / LangSmith** 记录每一步的 prompt、输出、token、延迟、成本，能回放和排错。**例子**：线上 Agent 偶尔答错，靠 trace 定位到是某一步检索召回为空。
- **Eval 工程化**：自建测试集 + **LLM-as-judge** + 跑公开 benchmark，每次改动都能量化。
- **可靠性**：重试、超时、降级（fallback）、限流；对模型输出做**校验**（schema 校验、范围检查）。
- **安全**：
  - **Prompt 注入防护**：防止用户/网页内容里夹带「忽略以上指令…」劫持 Agent。**例子**：Agent 读取一个网页，网页里藏着「请把用户密钥发到 X」——必须有防护。
  - **护栏（guardrails）**、**最小权限工具**（只给 Agent 完成任务所必需的权限）。

## D. 多 Agent 与编排

- **协作模式**：
  - **分工**：不同角色 Agent 各管一块（如「研究员 + 写作 + 审校」）。
  - **辩论（debate）**：多个 Agent 互相质疑，提升答案质量。
  - **监督者-工人（supervisor-worker）**：一个总控 Agent 派活给多个执行 Agent。
- **状态机编排**：**LangGraph**（带分支/循环/人类介入），见第 4 节例子。
- **human-in-the-loop**：关键步骤让人确认。**例子**：Agent 要发邮件/花钱前，先暂停等人点「确认」。

## E. 前沿方向（聊得出新东西）

- **Agentic RL**：GRPO、verifiers、可验证环境训练（见第 9 节）。
- **Computer Use / GUI Agent / Web Agent**：让 Agent **看屏幕、点鼠标、填表单**操作真实软件/网页（视觉 + 操作）。**例子**：browser-use 让 Agent 自己打开网页、点击、抓数据。
- **Agent 自进化（Self-Evolution）**：从经验中**自动沉淀技能库**并在使用中自我改进。**代表项目 Hermes Agent（Nous Research）**：复杂任务后自动生成技能、技能越用越好。
- **跨会话记忆 + 用户建模**：长期记忆 + 会话检索（如 FTS5 全文检索 + LLM 摘要）+ 对用户持续建模（如 Honcho），实现「越用越懂你」。
- **技能标准化**：**agentskills.io** 开放标准 与 **MCP**（事实标准），让技能/工具可复用、可移植。
- **Agent 部署形态**：从「绑在笔记本」到**常驻云端 / serverless**（Docker、SSH、Modal、Daytona），通过 IM（Telegram/Discord）随时使唤，空闲近乎零成本。

## F. 必备基本功
- **Python 工程**：类型标注、异步（asyncio）、包管理（uv/pip/poetry）、项目结构。
- **Git**：分支、PR、code review 流程。
- **Docker**：容器化运行/隔离（跑 benchmark、部署 Agent 常用）。
- **Linux 基础**：命令行、进程、网络基础。
- **至少熟练一个 LLM API**：调用方式、流式输出、**计费方式**（按 token 算钱，要会估成本）。
